Device QA — Offline-sync robustness (Wave 15)

Purpose: verify the three targeted fixes that close real failure modes the MVP hit during Wave 5 QA: (A) silent upsertClient failures that left ghost ops in pending_ops, (B) no self-recovery when the server revokes the session (session_not_found / 403), (C) no manual escape hatch when the queue is stuck.

In scope: null-return on upsert_client_with_id now throws UpsertClientNullResultException and routes through the flush catch block (attempts increment, stale-op classifier applies, drops at 30 attempts); ApiClient _guardAuth funnel catches session_not_found, forces local sign-out, flips sessionExpired on AuthService; coral banner on Home + Studio with "Sign in" CTA; Diagnostics has a Force sync now action row with a toast; polish items: pullAll fires on signedIn and a 5s retry backoff skips recently-failed ops.

Out of scope: dropped-op audit log, orphan cache cleanup, amber/red pending-chip state (queued as Wave 15.5); any Wave 4 Phase 2 / line-drawing / web-player work; iOS Swift changes; Wave 14 /members files.

Prerequisites

  • Main merged through PR for Wave 15. feat/wave-15-offline-sync-robustness branch landed.
  • Install to device via ./install-device.sh from the repo root. The PR did NOT ship a device install; Carl needs to reinstall before running this script.
  • At least two practices you're a member of — one for the main test path, one to craft a mismatched-practice upsert (item 1).
  • Supabase CLI access for the session-revocation simulation (item 3 — supabase db query --linked "DELETE FROM auth.sessions WHERE user_id = '<your-uuid>'"). Alternatively rotate your password from a second device.

1 · A — Silent upsertClient failure now recovers

Null-return from upsert_client_with_id used to dead-end the flush loop without incrementing attempts or firing any dev.log. Now it throws and rides the normal error path.

  • 1. Force a null-return by crafting an upsert op with a practice_id you aren't a member of (set SyncService.instance.storage.db.rawInsert via a debug console, or by swapping the current practice in SharedPreferences to a UUID that doesn't exist). Watch os_log (idevicesyslog or Console.app) — each retry should emit a SyncService: flush upsertClient id=… code=null: UpsertClientNullResultException line. Attempts count increments per retry (check Settings → Diagnostics → Pending ops queue after ~5 retries).
  • 2. Drive the crafted op past 30 attempts (tap Force sync now in Diagnostics repeatedly, or let the normal offline→online triggers accumulate). After attempts ≥ 30, the op drops from pending_ops. dev.log emits flush upsertClient id=… dropped (attempts=30 exceeded cap). Diagnostics pending-ops row goes back to 0 (or drops by the count of the stuck op).

2 · Drain-on-boot still fires (sanity from Wave 14)

Confirm the existing boot drain wasn't broken by the retry backoff.

  • 3. Enqueue a legit client-rename op offline (airplane mode ON, edit a client name in Home), force-kill the app before the queue drains, re-enable network, cold-launch. Within ~5s of boot, os_log shows SyncService: boot — 1 pending ops in queue followed by online transition or the auth-state listener firing a flush. Client name on server updates without any user action.

3 · B — Session-not-found self-recovery

Before Wave 15, a server-side session deletion produced silent 403s forever. Now the app detects it on the next RPC and surfaces a coral banner.

  • 4. Signed in on the device, delete your auth.sessions row from the Supabase side. Run supabase db query --linked "DELETE FROM auth.sessions WHERE user_id = '<your-uuid>';" (grab the uuid from Diagnostics → Context → Practice first, or via SELECT id FROM auth.users WHERE email = 'yours@example.com';). Within one RPC round-trip (pull clients / credit balance / rename something), the coral banner appears at the top of Home: "Your session expired. Please sign in again to continue syncing."
  • 5. Swipe into Studio mode while the banner is up — the same banner appears at the top of Studio. os_log shows ApiClient: session revoked — forcing sign-out exactly once (re-entry guard works; no spam).
  • 6. Tap the Sign in CTA on the banner → AuthGate routes to the sign-in screen (ApiClient has already called auth.signOut() server-side). Reads from SQLite cache still work between steps 4 and 6 — the app isn't force-navigated mid-action.
  • 7. Sign back in with password or magic link. After the onAuthStateChange(signedIn) event lands, the banner vanishes on the next frame. Queue flushes automatically (os_log: auth signedIn — draining pending ops). A pullAll fires for the last-known practice (os_log: auth signedIn — refreshing cache for ≤practice-id>).

4 · C — Force sync now escape hatch

Settings → Diagnostics now carries a visible manual drain action.

  • 8. Open Settings → Diagnostics. Below the Context card you see a new ACTIONS section with a "Force sync now" row (sync icon, "Drain the pending-ops queue immediately" subtitle, chevron right). Card chrome matches the Boot probes card above.
  • 9. With the queue non-empty (enqueue a rename offline first), tap Force sync now. Haptic click fires. Toast reads "Flushed N ops · M remain" with real counts. If you tap again after a successful drain, toast reads "No pending ops to sync." Pending-ops probe row above updates to reflect the new depth.
  • 10. Button is always enabled (not greyed on queue-empty). Useful as a heartbeat when Diagnostics shows 0 but os_log suggests otherwise — tap it, see either a flush happen or a "No pending ops to sync." toast confirming the empty state.

5 · Polish — retry backoff

Rapid enqueues within 5s no longer hammer the server with duplicate retries of the same op.

  • 11. Prime a failing op (e.g. the crafted mismatched-practice upsert from item 1, which throws null-result). Within 5s of its first attempt, fire 3 more Force sync now taps back-to-back. os_log only shows ONE flush upsertClient id=… failed line per 5s window — the intermediate flush() calls saw lastAttemptAt inside the cool-down and skipped the op. After ~5s, the next Force sync now retries and logs again.
