Device QA — Editor sheet opens with content (LateInit fix)

P0 unblock. Carl's iPhone (staging tip 6226675) opened the exercise editor sheet as a blank grey translucent sheet with no content. Blocked all editing flows. This script verifies the forward-fix in PR #341.

Purpose: verify the editor sheet renders its full content (tab strip + Demo tab video + chrome) when opened from a Studio exercise card, and that the Demo-default + continuous swipe-wrap feature from PR #340 still works exactly as designed.

Bug being fixed: PR #340's commit 13b8c59 declared _pageController as a late final field assigned via _pageController = PageController(initialPage: _pageForCell(...)) in initState. But _pageForCell() itself reads _pageController.hasClients to choose between the midpoint-anchor and nearest-cycle math — and it's called before the assignment lands, throwing LateInitializationError. Flutter's release-mode error widget silently replaces the subtree with an invisible ErrorWidget, leaving the bottom sheet's grey backdrop chrome visible with no content.

Fix shape: added a bool _pageControllerInitialized = false; field alongside _pageController. Flipped true the instant the PageController assignment lands in initState. Modified _pageForCell() to gate the _pageController.hasClients read behind _pageControllerInitialized && _pageController.hasClients so the pre-assignment path takes the midpoint-anchor branch instead of throwing.

Invariant reinforced: late final fields can't be touched until assigned. If initState needs a helper that reads the field, the helper must check an explicit initialization guard — hasClients alone is not enough.

Out of scope: the Demo-default + swipe-wrap behaviour itself (already covered by item 0 2026-05-15 script — re-run that once this P0 verification passes). Studio card chrome, lobby, web player.
