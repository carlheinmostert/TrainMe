Mobile QA — hero crop alignment on Studio / ClientSessions / Camera peek / Preview

Scope: consume the hero_crop_offset column shipped in PR 1 across every practitioner-facing thumbnail surface. The same data drives the web-player lobby in PR 4. Math: offset in [0..1] → Flutter Alignment(x, y) on BoxFit.cover; landscape sources slide the crop along X, portrait sources slide it along Y, square sources ignore the value.

Behaviour contract: NULL hero_crop_offset (every legacy row + every fresh capture until PR 2 ships and the practitioner authors a value) falls through to 0.5 — the existing centred crop. So all of these surfaces should be pixel-stable against pre-PR-3 builds for any plan whose exercises are still NULL. Visible differences only appear once PR 2 lands and a practitioner authors a non-centred crop.

Pre-condition for items 1–5: at least one published plan with at least one video exercise whose hero_crop_offset is set to a non-default value. Until PR 2 ships the editor, the cleanest way to do this is via the Supabase SQL editor: UPDATE public.exercises SET hero_crop_offset = 0.10 WHERE id = '<exercise-id>'; then force a mobile sync (kill + reopen).

Pre-flight

  • PR 1 (hero_crop_offset schema + plumbing) is on main + the migration is applied.
  • Mobile re-install via ./install-device.sh picks up the PR 3 consumer changes.
  • For non-default-crop testing: at least one published plan with at least one video exercise whose hero_crop_offset is set to a clearly non-centred value (e.g. 0.10 or 0.85). PR 2's editor is the eventual authoring path; until it ships, set the value via SQL editor + force sync.

  • 2. Studio exercise card (Hero frame) honours the authored crop. Tap the session card to open Studio mode. The list of exercise cards renders the static Hero frame as a flood-fill background.   • 3. Editor sheet header honours the authored crop. From Studio, tap an exercise card to open the editor sheet. 3 · Orientation handling (landscape vs portrait sources)

  • 8. Landscape source — offset slides along X. Pick / capture a landscape exercise (camera rotated, wider than tall after rotation). Set hero_crop_offset = 0.0 via SQL & sync. The Studio card, filmstrip cell, and editor header should all show the LEFT edge of the source (its widest crop). Set to 1.0, sync — same surfaces show the RIGHT edge. The Y axis stays centred (the constrained axis hugs the full short edge).
  • 9. Portrait source — offset slides along Y. Pick a portrait exercise (taller than wide, the most common iPhone capture). Set hero_crop_offset = 0.0 via SQL & sync. The Studio card, filmstrip cell, and editor header should all show the TOP edge of the source. Set to 1.0, sync — same surfaces show the BOTTOM edge. The X axis stays centred. Visually: a portrait exercise where the practitioner squats then stands — 0.0 shows the head/standing pose, 1.0 shows the feet/squat pose.
  • 10. Unknown aspect (aspect_ratio = NULL) falls through to centred. Find / fabricate a row whose aspect_ratio is null (extremely rare — only failed-conversion-probe rows). Even with a non-default hero_crop_offset, the consumer surfaces render centred (we can't pick an axis without orientation). No crash; no warning.
