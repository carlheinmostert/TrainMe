2026-04-24 · Wave 20 · Soft trim

Device QA — Wave 20 soft trim

Purpose: verify the new per-exercise soft-trim feature end-to-end across schema, mobile editor, mobile playback, web player, and persistence.

What landed: a practitioner-controlled in/out window per exercise. Stored as start_offset_ms + end_offset_ms on the exercise row. Playback (mobile preview AND web player) clamps to the window and loops within it. NO re-conversion — the underlying media file stays full-length. Same trim applies across all three treatments (Line / B&W / Original) since they share source timing.

Surfaces touched: Supabase migration (schema_milestone_x_soft_trim.sql; get_plan_full auto-includes the new keys via to_jsonb(e); replace_plan_exercises extended), mobile ExerciseCapture (schema v25), Studio _MediaViewer (new bottom trim panel + position-clamp listener), web-player bundle (delegated loadedmetadata + timeupdate handlers).

SW cache name: homefit-player-v43-soft-trim (was v42-vertical-chrome). First Safari visit after install should show the new name.

Punted: long-press 4× zoom on the trim panel handles (drag affordance alone in v1).

Prerequisites

  • Latest ./install-device.sh (today). Build SHA on Home footer should match the Wave 20 commits.
  • Safari hard-reload on any plan URL so the service worker advances to homefit-player-v43-soft-trim.
  • A capture that's at least 5 s long so trimming is meaningful — fresh capture today is fine.

A · Schema migration round-trip

  • 1. Supabase columns + RPC body include the new fields.
    1. Confirm start_offset_ms + end_offset_ms exist on public.exercises (columns: integer, nullable). Already verified at apply-time on the linked project.
    2. Capture a fresh video, set a trim window in Studio (any in/out you like), tap Publish.
    3. In Supabase Studio: SELECT id, start_offset_ms, end_offset_ms FROM exercises WHERE plan_id = '<your-plan-id>'; — expect both columns populated.
    4. Open the plan URL in Safari (or in the practitioner preview WebView); Web Inspector → Network → get_plan_full response — expect start_offset_ms + end_offset_ms on each exercise dict.

B · Mobile trim-editor UI

  • 2. Trim panel is visible only on video exercises ≥ 1s.
    1. In Studio, tap a VIDEO exercise's thumbnail → _MediaViewer opens fullscreen.
    2. Expect: bottom-anchored trim panel (~96 px tall) above the safe area. Two coral handle pills + greyed bookend track + coral fill band + tick row + readout + (initially) NO reset link.
    3. Open a PHOTO exercise viewer — expect: NO trim panel. Bottom chrome (mute / dots / play-pause) sits at the original positions.
    4. Open a REST row from the deck (if any) — expect: viewer doesn't open at all (rest is no-op). Confirms the panel is hidden for rest semantics.
  • 3. Drag both handles updates the window live.
    1. Drag the LEFT (start) handle to the right ~30% of the bar.
    2. Expect: coral fill shrinks from the left; greyed bookend grows on the left; tick-row middle readout updates live (m:ss–m:ss).
    3. Expect: a "Trimmed: m:ss · N reps loops in m:ss" readout under the bar updates live; "⤺ Reset to full video" link appears.
    4. Drag the RIGHT (end) handle to the left ~70% of the bar.
    5. Expect: coral fill shrinks from the right; right bookend grows; readouts update live.
    6. Reset to full video link → tap → expect: both handles snap to the bar ends; the link vanishes.
  • 4. Minimum 0.3 s window guard — handles cannot cross.
    1. Drag the LEFT handle as far right as it will go.
    2. Expect: it stops 0.3 s short of the right handle's position; you should feel a light haptic pulse when it bumps the guard. The middle tick-row readout's gap stays at 0:00-ish but never inverts.
    3. Reverse: drag the RIGHT handle as far left as it will go — same behaviour against the left handle.
    4. Reset to full video.
  • 5. Tap inside coral range scrubs; tap in greyed bookend is no-op.
    1. Set a trim window of roughly the middle 50% of the bar.
    2. Tap once inside the coral fill. Expect: video currentTime jumps to that position (the visible frame changes).
    3. Tap once inside the LEFT greyed bookend. Expect: nothing happens — no scrub, no visible jump (those frames don't exist in the trimmed slice).
    4. Same for the RIGHT bookend — no-op.

C · Persistence + treatment switching

  • 6. Trim survives close + reopen of the viewer.
    1. Set a trim window (any in/out).
    2. Tap the Close (X) button top-right.
    3. Re-tap the same exercise's thumbnail to reopen the viewer.
    4. Expect: same trim window — handles where you left them; same readouts; "⤺ Reset to full video" link is showing.
    5. Force-quit + relaunch the app, reopen the viewer — expect the trim is still there (SQLite v25 persistence).
  • 7. Switching treatment via the vertical pill does NOT reset trim.
    1. Set a trim window. Note the handle positions.
    2. Cycle to B&W via the left vertical pill (or vertical swipe). Expect: video swaps to the grayscale treatment; trim handles stay where they were; coral fill unchanged.
    3. Cycle to Original — same: trim unchanged.
    4. Cycle back to Line — same: trim unchanged.
    5. Confirms the trim is per-exercise, not per-treatment.

D · Mobile preview playback

  • 8. Mobile _MediaViewer playback respects the trim.
    1. Set a clear trim window — say roughly seconds 1–3 of a 5 s capture.
    2. Watch the video play. Expect: it never shows frames before the start handle; loops back to the start handle when it crosses the end handle.
    3. Reset to full video → expect playback resumes the FULL clip end-to-end (no clamp).
    4. Set the start handle in the middle and the end handle at the right edge → reopen the viewer → expect first painted frame is at the start-handle position, not at 0:00.

E · Web-player playback

  • 9. Web player respects the trim end-to-end.
    1. Publish the session that has the trim window set.
    2. Open the plan URL in Safari (hard-reload first so SW caches v43-soft-trim).
    3. Expect: <video> auto-plays from the trim start; loops back to start when it crosses the trim end. Same for B&W + Original treatments (per-exercise preferred_treatment drives which one renders).
    4. Web Inspector → Network → confirm the get_plan_full response includes start_offset_ms + end_offset_ms on the trimmed exercise; null on untrimmed ones.
    5. Expect: NO trim editor visible on the web player — editing is mobile-only. The web player only respects existing metadata.

F · Backwards compatibility

  • 10. Legacy exercises (null offsets) play the full clip on both surfaces.
    1. Open a session that pre-dates Wave 20 (any older capture you haven't trimmed). Expect: trim panel is still visible (videos always get one once they're past 1 s) but no handles are pulled in — both at the bar ends; no "⤺ Reset" link.
    2. Play the video → expect: full clip, end-to-end, native loop. Same on the web player after publish.
    3. Confirms null offsets fall through to "no clamp" — pre-feature behaviour preserved.
