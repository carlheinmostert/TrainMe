2026-05-16 · service-worker freshness fix

Device QA — SW network-first for app shell

Production-critical fix. The previous service worker used cache-first for the entire app shell (HTML / JS / CSS / config), which meant new deploys did not propagate to clients with an active SW — even after a hard reload or Safari Private Browsing. Carl hit this when PR #370's fix was deployed but his iPhone Safari kept serving the older v70-png-modal-removed bundle.

Root cause. A client's first visit installs the SW + pre-caches the app shell. Subsequent visits route through the SW's cacheFirstStrategy, which returns the cached bytes without checking the network. Even when the SW updates in the background (skipWaiting + clients.claim) and the cache name changes to a fresh SHA, the CURRENT page keeps the old JS in memory until every tab closes and reopens. Safari amplifies this by holding the old SW across Private Browsing sessions.

Fix. Three-way routing in the SW fetch handler:
  • Supabase API → network-first (unchanged).
  • Media files (mp4 / jpg / png at /storage/v1/object/public/media/) → cache-first (unchanged — URLs are content-addressable, immutable).
  • Everything else (app shell, lobby.js, future assets) → network-first with cache fallback for offline. New deploys propagate on next reload; offline still works via the cached copies.

Trade-off. ~50–200ms added per asset on cold reload when online (the network roundtrip previously skipped via cache). Acceptable vs. the alternative of clients stuck on stale code.

R-10 parity. web-player/sw.js and app/assets/web-player/sw.js byte-identical post-edit.

  • Branch fix/sw-network-first-app-shell deployed to staging.session.homefit.studio (Vercel auto-deploy) AND a fresh Flutter install on Carl's iPhone via ./install-device.sh staging.
