2026-05-15 · P1 wiring fix on PR-C · reactive failures

Device QA — Publish sheet “Show which files” tap-target

Bug: on the failure state of the publish progress sheet (PR-C), the “Show which files →” tap-target was invisible. The Flutter wiring captured _publishFailures at sheet-construction time when it was still const []; by the time uploadPlan() returned with the populated optionalArtifactFailures list, the sheet widget had already captured the empty list as a prop. The conditional widget.failures.isNotEmpty was therefore always false on the failure render — the tap-target never appeared.

Fix: extend PublishProgress.failure to carry a List<UploadFailureRecord> failures field. The upload service emits the failures on the same stream event that flips failed=true. The sheet now reads p.failures from the stream snapshot, not from the captured prop. The out-of-band setState path in studio_mode_screen.dart is retained so the retry chip's tap-through (which re-opens the sheet against an older event) still has a fallback list to hand to the diagnostic sheet.

Scope: repro requires a failure mid-upload — same setup as items 7–10 of the PR-C QA, but the focused expectation here is that the tap-target is VISIBLE the instant the failure state renders, not delayed or absent.

Prerequisites

  • Build from fix/publish-sheet-failures-reactive installed on Carl's iPhone via ./install-device.sh.
  • One fully-consented test client (B&W + Original ON) with a fresh session of 3+ video captures.
  • Wi-Fi reachable and Control Center one-tap away so Airplane mode can be flipped mid-upload.

Repro · failure with visible “Show which files”

  • 1. Open the prepared session. Tap PUBLISH on the workflow pill. Sheet opens on the Publishing plan state with five phase rows. Confirm Uploading treatments reaches the active (pulsing coral) state and the N of M files subtitle is ticking up. device
  • 2. Flip Airplane mode ON from Control Center while Uploading treatments is still active. Within ~10s the upload fails. Expect: row turns coral with !; header morphs to coral “Publish failed”. device
  • 3. The fix. On the same frame the failure state appears, the “Show which files →” coral-underline link must be VISIBLE above the Retry publish button. No delay, no need to dismiss-and-reopen, no need to wait for a second event. Compare against the bug: previously this row was invisible on the first failure render. device
  • 4. Tap “Show which files →”. Expect: the existing UploadDiagnosticSheet opens with one row per failed variant (kind chip, exercise #N + name, storage path, local path, exists-on-disk). Copy all works. Close the diagnostic sheet and you return to the failure-state progress sheet underneath. device

Re-open path · tap chip after swipe-dismiss

  • 5. With the failure sheet still open from item 3, swipe it down to dismiss. Toolbar should show the failure chip. Tap the chip to re-open. Expect: the failure sheet re-opens with the SAME failure list visible — “Show which files →” still present, retry button still present. This confirms the prop fallback path (host-side _publishFailures) works when the sheet re-attaches to a notifier holding an older event. device
  • 6. Toggle airplane mode OFF. Tap “Retry publish”. Expect: sheet pops, fresh sheet opens with rows resetting, publish re-fires and succeeds (all rows green, auto-dismiss after the “All set” beat). No leaked failure state from the previous attempt. device
