Wave 35 · device QA

Portal toaster top + role persistence + member remove + Settings stats + Preview focus

Five mixed-surface fixes from Wave 5+14 device QA results plus one new Studio handoff feature. (A) Portal toaster moves to top-of-page across every surface (was bottom-6, getting clipped on mobile Safari + tall pages). New shared Toast.tsx component used by /members add-feedback; the rest of the portal got a className swap from bottom-6 to top-4. (B) Role-change dropdown now persists on refresh (W5+14 #15). /members page also pinned to force-dynamic so Next.js 15 can't replay a stale RSC payload after the RPC commits. Self-demote disabled + last-owner block now re-testable as a side-effect. (C) Remove member no longer errors with relation "public.practice_invite_codes" does not exist (W5+14 #18). Wave 14 dropped the table; the Wave 35 migration (schema_wave35_drop_invite_codes_ref.sql) drops the dead UPDATE inside remove_practice_member. (D) Settings → Network: stats row removed, Network section header + invite copy + share widget + portal hand-off CTA all kept (W5+14 #20 reinterpretation). (E) NEW — Preview → Studio focus handoff. Closing the media viewer scrolls the just-viewed exercise into view + auto-expands the matching card + adds a subtle coral border / elevation lift. Clears on next interaction.

A · Portal toaster top-of-page

  • 1. Members — Add by email toast pops at TOP of page (was bottom).
    1. On manage.homefit.studio/members, scroll the page down so the Add form is just barely visible at top.
    2. Type any email in the Add field + click Add. (Use a never-seen address so it lands as pending.)
    3. Toast slides DOWN from top-center with the "Queued for signup" headline + "Saved — will join automatically when they sign up." body. Auto-dismisses after ~8s. Same toast colour palette as before (coral border + raised surface), only position changed.
    4. Re-test on iPhone Safari with the page scrolled fully down — toast still visible (was previously clipped off bottom).
  • 2. Other portal toasts also top-positioned (Clients delete, Sessions delete, practice rename, share-kit copy).
    1. On /clients, swipe-delete a client. Undo toast appears at TOP center.
    2. On /network open share-kit and copy something. Confirmation toast at TOP center.
    3. Anywhere with a "rename practice" / "rename client" affordance — rename. Confirmation toast at TOP center.
    4. No surface still pops at the bottom. (If you find one, capture the URL in the note.)

B · Role-change persistence + cascading guards

  • 3. Promote a practitioner to owner via the Role dropdown — change persists on refresh.
    1. On a practice with at least one practitioner besides yourself, open /members.
    2. Click the Role dropdown for the practitioner. Pick OWNER.
    3. Toast confirms "<Name> is now an owner." at top of page.
    4. Cmd+R / hard refresh. Their role still reads OWNER. (Was the W5+14 #15 fail — UI showed change but reverted on refresh.)
    5. Demote them back to practitioner. Toast + persist on refresh.
  • 4. Self-row Role dropdown is missing (you can't change your own role).
    1. On /members, find your own row (tagged "you").
    2. The Role column shows the role as a STATIC chip ("OWNER" / "PRACTITIONER"), not a dropdown. The DB-level rejection ("cannot change your own role") doesn't need to fire because the UI never lets you try.
  • 5. Last-owner demote is blocked with a usable error toast.
    1. Set up a practice where you are the ONLY owner + at least one practitioner exists.
    2. Promote a practitioner to owner. Verify role persists (item 3 covers).
    3. Now demote the OTHER owner back to practitioner (you're not the last owner yet, you have 2). That should succeed.
    4. Try to demote the OTHER owner again from owner → practitioner. Wait — you can't, they're already practitioner. Try yourself: there's no dropdown on your own row (item 4). So the last-owner block is implicit through the self-demote restriction.
    5. If you ever DO get a last-owner block to fire (e.g. by promoting + demoting in sequence), confirm the toast reads "cannot demote the last owner" or similar PostgreSQL-message at TOP of page (red error tone), not a 500.
  • 6. Role change toast colour is success (coral) not error (red).
    1. Promote / demote a practitioner. Toast at top should have the coral / brand-accented border + heading, not the red error tone.

C · Remove member no longer errors

  • 7. Click Remove on a practitioner row — succeeds without a 500.
    1. On /members, find a practitioner you can remove (not yourself).
    2. Click the Remove text button at the right of the row.
    3. Row disappears optimistically. Top-of-page toast confirms "<Name> removed." (NOT a red error toast saying relation "public.practice_invite_codes" does not exist).
    4. Cmd+R / hard refresh. Row stays gone.
    5. If their session is still open in another tab/device, they get booted out of that practice on next portal action (covered by W5+14 #18 originally, this just verifies the Wave 14 regression is fixed.)

D · Settings → Network stats row removed

  • 8. Mobile Settings → Network section: stats row gone, rest intact.
    1. Open the trainer app → Settings.
    2. Scroll to the "Network" section header.
    3. Section now shows: header, "Your practitioner network" copy + invite tagline, share-and-copy widget (referral code pill + Share invite link button), and the "View full network on the portal →" CTA. THAT'S IT.
    4. The 4-tile stats row (Network rebate / Rebate lifetime / In network / Network spend) is GONE. No stub, no empty placeholder, just clean — the divider above the portal CTA is also gone since there's nothing to separate from.
    5. "View full network on the portal →" still taps through to manage.homefit.studio/dashboard.

E · Preview → Studio focus handoff (NEW)

  • 9. Closing Preview scrolls the last-viewed exercise into view + auto-expands its card.
    1. Open a session with at least 8 exercises so the list is scrollable.
    2. In Studio, scroll to the OLDEST exercise (top of the bottom-anchored reverse list). Tap its thumbnail to open the media viewer ("Preview").
    3. Inside Preview, swipe RIGHT (or tap the next chevron) several times until you've moved to a NEW exercise that wasn't visible in your Studio scroll position.
    4. Tap the × close button (top-right of Preview).
    5. Studio re-appears with the just-viewed exercise's card scrolled into view (centered) AND auto-expanded (coral perimeter, 2px border, accordion ready).
    6. A subtle coral elevation shadow surrounds the focused card — reads as "lifted" not "boxed in".
  • 10. Auto-expand fires the SAME behaviour as tapping the card header (not a new mode).
    1. Repeat item 9. After Preview closes, the focused card is expanded.
    2. Inside the expanded card, the DOSE / PACING / NOTES accordion groups are all CLOSED (the spec default for a freshly-expanded card). No PLAYBACK group either — W29 retired it.
    3. Tap PACING in the focused card — it opens normally. Tap PACING again — closes.
    4. Tap the card header itself — collapses the card (becomes the "I came back here, I'm done" gesture).
  • 11. Focus marker clears on next user interaction.
    1. Trigger the focus handoff (item 9). Card has the elevation shadow.
    2. Scroll the list. Elevation shadow disappears (border collapse to 1px hairline once the card collapses on a different card's tap; or, if you just scroll without tapping, the shadow drops the moment the scroll moves more than ~4px).
    3. Trigger again. Tap a different exercise card. Focus shadow drops. The newly-tapped card expands as usual.
    4. Trigger again. Edit a field on the focused card (reps / sets / hold). Focus shadow drops; the card stays expanded so the edit can complete.
    5. Open a fresh session. No stale focus marker on any card.
  • 12. System back gesture (iOS swipe-from-left-edge) ALSO triggers the focus handoff.
    1. Trigger Preview as in item 9; swipe to a new exercise.
    2. Instead of tapping the × close button, swipe from the LEFT screen edge inward (iOS back gesture).
    3. Studio appears with the same scroll-into-view + auto-expand behaviour as the close-button path. Focus shadow visible.
    4. If swipe-back isn't possible on this build (gesture intercepted by something), capture in note. The close-button path (item 9) is the canonical one and must always work.
