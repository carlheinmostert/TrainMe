2026-05-15 · BUG 13 follow-up · in-app upload diagnostic surface

Device QA — Publish toast: in-app diagnostic sheet

Purpose: verify the new in-app diagnostic surface that fires when the post-publish toast reports “Some optional treatment files are still processing”. Builds on PR #335's debugPrint logging — instead of needing Xcode device console access, tapping Details on the toast now opens a bottom sheet listing each failed file (kind, exercise, exists-on-disk, bucket path), plus a Copy all action that puts a paste-ready diagnostic on the clipboard.

Scope: diagnostic-only patch. No behavioural change to the upload retry path. The bottom sheet is read-only; it doesn't trigger a retry — the next publish does that automatically.

Out of scope: the root-cause fix for BUG 13 itself. This PR exists so Carl can capture WHICH file(s) are failing on subsequent publishes and feed that back without needing a Mac.

Prerequisites

  • Build from fix/publish-toast-diagnostic-sheet installed on Carl's iPhone via ./install-device.sh.
  • A session that has previously been published and reproduces the “optional treatment files still processing” toast on re-publish (Carl's staging build has been reliably hitting this).
  • If no failing session is handy: capture a fresh photo + video session, publish, then re-publish — the toast almost always fires at least once in the test environment.

1 · Toast still appears, now with a Details action

  • 1. Publish a session that triggers the partial-upload path. The existing “Published. Some optional treatment files are still processing…” SnackBar still appears (floating, dark, white text). Copy is unchanged from before this PR. device
  • 2. A coral Details action sits at the right end of the SnackBar (next to the message body). Duration of the toast is long enough to actually tap the action (~12s, was 8s) — if it dismisses before you can tap, that's a regression. device

2 · Diagnostic sheet opens

  • 3. Tapping Details opens a bottom sheet titled “Upload diagnostic — last publish”. The sheet is dark (surfaceRaised), has a drag handle, and lists one card per failed file. The subtitle row reads “N optional file(s) failed to upload. Main plan published OK; retry publish to backfill.”. device
  • 4. Each failure card shows: a coral chip with the kind (e.g. raw_archive_color_thumb_failed, raw_archive_segmented_upload_failed); a sage/red exists/missing indicator with icon, reflecting whether the local source file was present at the time of failure; the exercise label as “Exercise: #N {name or id}”; the bucket storage path; and the local-path (relativised to Documents/… — no sandbox UUID leak). The bucket + local paths are selectable text so you can pinch-zoom and copy-fragment if needed. device

  • 5. Tap Copy all at the bottom of the sheet. A confirmation SnackBar appears: “Copied N failure record(s) to clipboard”.   • 6. Tap Close (the grey outlined button next to Copy all) — the sheet dismisses cleanly. Swipe-down on the drag handle also dismisses. Returning to Studio shows the normal post-publish state — no leftover overlays / scrim. device
