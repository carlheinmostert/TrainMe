Schema QA — hero_crop_offset column + RPC plumbing

Scope: data plumbing only. Adds exercises.hero_crop_offset (NUMERIC nullable) on Supabase + the matching SQLite column (REAL nullable, schema v37 → v38), wires it through replace_plan_exercises + the upload payload + the offline pull mapper + the unified preview scheme bridge. No UI — no editor changes, no Studio-card consumers, no web-player surface. Those are PR 2 (editor), PR 3 (mobile thumbnail consumers), PR 4 (web-player lobby).

Behaviour contract: NULL hero_crop_offset (every legacy row + every fresh capture until a practitioner touches it) means “consumer uses 0.5 (centred crop)”. So this PR is intentionally byte-stable on the wire and pixel-stable on every existing surface.

Surprise found mid-PR: the prior Wave Hero migration (focus_frame_offset_ms) shipped the local-write path + upload payload + scheme bridge but never added the pull branch in SyncService._exercisesFromPlanRpc. Backfilled here so both Wave Hero and Wave Lobby round-trip cleanly through the offline cache — verify item 6 covers this.

Prerequisites

  • Carl applies the migration via supabase db query --linked --file supabase/schema_wave_hero_crop.sql after review.
  • Mobile re-install via ./install-device.sh picks up SQLite v38 + the new column on the upload + sync paths.
  • At least one published plan with at least one video exercise on the device, so the round-trip can hit a real row.

1 · Cloud schema

  • 1. Migration applies cleanly. Run supabase db query --linked --file supabase/schema_wave_hero_crop.sql. No errors. Idempotent — rerunning it on top is also a no-op.
  • 2. Column visible in information_schema. Run: 
    SELECT column_name, data_type, is_nullable
  FROM information_schema.columns
 WHERE table_schema = 'public'
   AND table_name = 'exercises'
   AND column_name = 'hero_crop_offset';
    Returns one row: hero_crop_offset | numeric | YES.
  • 3. replace_plan_exercises accepts hero_crop_offset. Run: 
    SELECT position('hero_crop_offset' IN pg_get_functiondef(
         'public.replace_plan_exercises(uuid,jsonb)'::regprocedure)) > 0;
    Returns t. Belt-and-braces: a manual replace_plan_exercises call with hero_crop_offset in the row payload (e.g. 0.32) persists the value — SELECT hero_crop_offset FROM public.exercises WHERE id = '<exercise-id>' returns 0.32.
  • 4. get_plan_full emits hero_crop_offset on each exercise. Run: 
    SELECT jsonb_path_query(
         public.get_plan_full('<plan-id>'::uuid),
         '$.exercises[*] ? (@.id == "<exercise-id>") . hero_crop_offset'
       );
    Returns the value set in item 3 (e.g. 0.32). For a fresh row that has never been touched the value is null — that's the expected legacy behaviour, NOT a fail.
