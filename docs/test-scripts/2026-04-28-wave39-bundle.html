Wave 39 · device QA

Sticky defaults + photo refresh + audit prepaid + plan opens

Four mixed-surface fixes. (A) Sticky per-client defaults (Wave 8) finally propagate forward on the next capture. The fix is a synchronous in-memory overlay layered on top of the SQLite cache: every override now lands in [StickyDefaults._memoryOverlay] before the async pending-op queue starts spinning. The next capture's prefill consults the overlay first, so a rapid edit-then-capture sequence inherits the override even if SQLite hasn't flushed yet. (B) Studio's conversion-event listener used to drop events whose row wasn't in _session.exercises yet — the typical race for a fast photo conversion captured as the LAST exercise. The listener now force-refreshes the session from SQLite (canonical store) when it can't find the row, so spinners clear in-place without leaving + reentering Studio. (C) Audit page now surfaces the unlock ↔ publish prepayment relationship: plan_issuances.prepaid_unlock_at is stamped on prepaid republishes, list_practice_audit threads it through, and the description column renders "Prepaid via unlock at …" on the publish row + "Used at …" / "Awaiting republish" on the matching unlock row. (D) Audit page now logs every plan-open as a sage plan.opened event with a NULL actor that renders as "Client". record_plan_opened RPC inserts the audit row alongside the existing first_opened_at / last_opened_at stamps. Schema migration: supabase/schema_wave39_audit_expansion.sql.

A · Sticky per-client default propagation

  • 1. Edit fields on exercise 1, capture exercise 2 on same client → pre-fills with overridden values.
    1. Open Client A. Create a new session. Capture a video (long-press shutter, ~3s). Wait for conversion to finish.
    2. Open the exercise card's DOSE accordion. Set reps = 12 (tap the 12 chip). Set sets = 4 (tap 4 chip). Set hold = 5s.
    3. Open the exercise card's PACING accordion. Set prep seconds = 10 (override input). Set DURATION PER REP → Manual → 6s.
    4. Swipe right to Capture mode IMMEDIATELY (don't wait). Long-press shutter and record exercise 2.
    5. Swipe back to Studio. Open exercise 2's DOSE accordion. Reps reads 12, sets reads 4, hold reads 5s. NOT the system defaults of 10 / 3 / 0s.
    6. Open exercise 2's PACING accordion. Prep seconds reads 10. DURATION PER REP's Manual seed reads 6s.
  • 2. Defaults are isolated per client — Client B's next capture uses ITS own override map, not Client A's.
    1. Carry the Client A defaults from item 1 (reps=12, sets=4, hold=5s).
    2. Back out to Home. Open or create Client B. Create a new session. Capture an exercise.
    3. Open the exercise card's DOSE accordion. Reps reads 10 (system default), sets reads 3, hold reads 0s. The Client A overrides do NOT leak.
    4. Edit Client B's exercise: reps = 8, sets = 2. Capture another exercise on Client B. New card opens with reps=8, sets=2.
    5. Switch back to Client A → create another new session → capture → reps=12, sets=4 (Client A's map preserved through the round-trip; not contaminated by Client B's 8/2).
  • 3. System-default fallback when no client default exists.
    1. Create a brand-new client (Client C) with no prior sessions. Open it → create session → capture exercise 1.
    2. Open the exercise card. Reps reads 10, sets 3, hold 0s — system defaults. (The cached_clients row exists with empty client_exercise_defaults jsonb; prefill is a no-op so withPersistenceDefaults fills 10/3/15.)
    3. Capture another exercise on Client C without touching any field. Same defaults (10/3/0s) — no override happened, nothing to write back.

B · Photo last-exercise conversion refresh

  • 4. Capture a photo as the LAST exercise — spinner clears in-place within ~1.5s.
    1. Create a new session on any client. Swipe right to Capture mode.
    2. Tap the shutter (short tap = photo). Photo captures. Swipe LEFT back to Studio immediately.
    3. Studio shows the photo card with a converting spinner. Wait. Within ~1.5s the spinner crossfades to a green check (conversion done). The card is now interactive — tap it → full-screen viewer opens.
    4. DO NOT navigate away and come back during the wait. The fix is that the spinner clears WITHOUT needing to leave Studio.
  • 5. Mid-list photo capture still updates correctly (regression check).
    1. In Studio, drag a photo card so it's NOT the last in the list (e.g. capture a video after, or reorder).
    2. Capture another photo — doesn't matter where in the list. Spinner appears, then clears within ~1.5s. Position in the list is correct (next available slot in capture order).
    3. Capture sequence: photo → video → photo → video. All four cards eventually settle to a green check. None get stuck on converting.

C · Audit prepaid-unlock relationship

  • 6. Publish row shows "Prepaid via unlock at …" subtitle when the publish consumed a prior unlock.
    1. On a published session whose 14-day grace has ELAPSED (or fake one with sqlite3 + ledger row), tap the padlock-style Unlock pill in Studio's bottom toolbar. Confirm the 1-credit unlock sheet.
    2. In the same session, edit something structural (add an exercise OR delete one). Hit Publish. Publish succeeds without charging another credit (consume_credit takes the prepaid fast path).
    3. Open the portal → Audit. Find the most recent plan.publish row. The Description column shows the plan title (line 1) PLUS a muted dim-grey second line: "Prepaid via unlock at {date}". The date matches the unlock row's timestamp.
  • 7. Unlock credit-consumption row shows "Used at {publish date}" OR "Awaiting republish".
    1. In the same Audit page, scroll to the credit.consumption row whose Description column reads unlock_plan_for_edit (the unlock from item 6).
    2. Subtitle line 2 reads "Used at {date}" — the date matches the matching plan.publish row's timestamp.
    3. Now tap the unlock pill on a DIFFERENT locked session. DO NOT republish — just leave it. Reload the Audit page. The new unlock row's subtitle reads "Awaiting republish" (no matching publish yet).

D · Audit plan opens

  • 8. Opening a published plan generates a plan.opened audit row.
    1. Take any published plan's share URL (https://session.homefit.studio/p/{uuid}). Open it in a fresh browser tab (or incognito so you're unauthenticated).
    2. Wait for the player to render the first slide.
    3. Refresh the portal's Audit page. The most recent row is a plan.opened chip with ts roughly = now. Description: "Client opened plan".
    4. Open the same URL again 10 minutes later (or just refresh). A NEW plan.opened row appears in the Audit log — it's logged on every open, not first-only.
  • 9. Chip tone is sage (engagement) and actor reads as "Client".
    1. On the audit page, locate any plan.opened row. The kind chip uses the SAGE colour (light green-ish background + emerald text) — the same palette as Credits-In events. NOT coral (which is the practitioner-publish tone).
    2. Actor column for the row reads "Client" in muted text (not "—", not blank). Anon web-player calls have no auth.uid().
    3. The Link column for the row points to the Player URL (same as plan.publish rows).
    4. Filter chip group: in the audit filter bar, expand "Credits in" (the sage group). plan.opened appears in that group's chip list. Toggle it on — only the engagement rows surface.

Out of scope — reminder

  • The 4 sticky-default fix is a Wave 8 follow-up. Field set is unchanged: reps / sets / hold_seconds / include_audio / preferred_treatment / prep_seconds / custom_duration_seconds (+ Wave 18.7 custom_duration_per_rep).
  • Photo conversion timing (~500ms) and dual-output pipeline (line drawing + segmented body-focus jpg) are unchanged. The fix is purely on the listener side in Studio: when an event arrives for a row not yet in _session.exercises, force a refresh from SQLite.
  • The audit page's unlock↔publish derivation is client-side — we walk the loaded page's rows once per render, build a Map keyed by meta.prepaid_unlock_at, and stamp the matching consumption row. With 50 rows per page, an unlock and its publish almost always co-locate; cross-page pairs would lose the "Used at…" subtitle (the publish's own subtitle still shows). Acceptable trade-off vs a server-side join.
  • plan.opened rows have NO trainer_id (anon writes). The actor filter dropdown can't target them; use the kind filter instead.
  • No mobile UI for any of the audit changes — portal-only.
