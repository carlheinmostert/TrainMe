QA — auto-register web-player assets in iOS scheme handler

Scope: app/ios/Runner/UnifiedPlayerSchemeHandler.swift. iOS embedded preview only — the public web player (session.homefit.studio) is unaffected.

Why this exists: PR #330 was a per-file hot-fix that added exercise_hero.js + jspdf.umd.min.js to a hardcoded if-else dispatch chain in webView(_:start:). The underlying brittleness — every new asset in app/assets/web-player/ needs an explicit Swift entry — remains.

Fix: data-driven asset resolver. The dispatch chain is now (1) config.js inert stub, (2) api/plan/<id> + local/<id>/<kind> dynamic routes, (3) generic asset lookup against the Flutter asset bundle (FlutterDartProject.lookupKey(forAsset:) + Bundle.main.path(forResource:)). Empty path falls back to index.html. Content-Type is derived from the filename extension. Unknown paths still 404 via the existing safeDidFail path.

Install: after merging the PR, install on Carl's iPhone via ./install-device.sh (Carl's usual workflow) or on the simulator via ./install-sim.sh. The change is a pure Swift refactor — no SQLite migration, no SW cache bump, no pubspec.yaml bump.

Prerequisites

  • Branch feat/scheme-handler-auto-register-assets merged into staging (or checked out locally + ./install-sim.sh).
  • The bundle already includes exercise_hero.js + jspdf.umd.min.js from the post PR #330 sync; no need to re-run tool/sync_web_player_bundle.dart.
  • Tests assume a published plan with at least one video exercise; if you don't have one ready, capture a 3-second video and publish before starting.

1 · Existing assets still serve

  • 1. Lobby loads — index.html at / is served. Open a published plan's preview from Studio. Expected: the lobby renders with the practitioner header + at least one exercise row + the Share-as-PDF button. Empty path (i.e. homefit-local://plan/) is mapped to index.html. If you see the WKWebView's native "Cannot Open Page" or the lobby is fully blank, this regressed.
  • 2. App boots — app.js + api.js served. From the lobby, tap an exercise row to enter the workout player. Expected: the player loads (countdown / rep-block stack / progress pills appear). If you see the lobby but tapping a row does nothing, app.js didn't load (check Console.app for [UnifiedPreview] NSLog entries).
  • 3. Styles applied — styles.css served. The lobby + workout-player surface should render with the dark coral homefit theme, not the unstyled-HTML default white background. If the page is light-grey / unstyled, styles.css failed to load.
  • 4. Hero rows render — lobby.js + exercise_hero.js served. On the lobby, each exercise row should show its Hero thumbnail (square coral-bordered image) on the left + exercise name + rep count. If rows are blank skeleton placeholders, the hero resolver script didn't load. This is the load-bearing case PR #330 hot-fixed; the new resolver should keep it working.
  • 5. PDF export — jspdf.umd.min.js + html2canvas.min.js served. On the lobby, tap the Share button. Expected: a PDF preview opens with the plan rendered as paginated PNG pages. If the button taps but nothing happens (or a console error logs about jspdf is not defined), one of the vendored scripts didn't load.

2 · Dynamic routes still match correctly

  • 6. Plan JSON loads — api/plan/<planId> route fires. The lobby calls fetch('api/plan/<planId>') early on; if this route is wrongly shadowed by the asset resolver, the page renders "Plan not found" or sits on the loading spinner forever. Expected: the lobby renders with the correct plan title + exercise count within ~2 seconds of entering the preview.
  • 7. Local media loads — local/<exerciseId>/<kind> route fires. Open a video exercise in the workout player. Expected: the converted video plays back. The route shape is homefit-local://plan/local/<id>/line (or /archive / /hero etc.) and is matched BEFORE the asset resolver, so video files never get mistakenly resolved as bundle assets.
  • 8. Unknown media kind still 404s. This is harder to drive from the UI; the existing parseLocalMediaPath rejects any kind outside the whitelist (line/archive/segmented/hero/hero_color/hero_line). Run the app + scrub Console.app for [UnifiedPreview] rejected unknown kind entries while interacting; expected: no rejections during normal playback. If you see them, a JS surface is asking for a kind the iOS guard doesn't accept.

3 · config.js still served as inert stub

  • 9. config.js returns the inert stub, not 404. The bundle's index.html includes <script src="config.js"> as the FIRST script tag. After this PR, the response should still be the literal text window.HOMEFIT_CONFIG = Object.freeze({}); (not a 404 and not a file from disk). How to verify: open the app, open Safari's Web Inspector against the embedded WebView, view the Network tab, and check the response body for config.js — should be the 1-line stub, ~52 bytes, served as application/javascript. If Carl can't easily attach the inspector, item 10 is the next-best proxy.
  • 10. Proxy for item 9 — no "not connected" / "Plan not found" on lobby open. If config.js were 404ing or serving a bad payload, api.js's isLocalSurface() path would still work (the inline fallback wraps the missing config), but the lobby loading state would log warnings. Expected: the lobby loads to the practitioner header + exercise list with no "Network error" / "No connection" screen.
