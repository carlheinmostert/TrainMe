Device QA — Hero Crop Resolver (web) + PDF Object-Fit Bug Fix

What landed: the web player's per-row hero image is now cropped to a 1:1 square via a single new module (web-player/hero_resolver.js) instead of CSS object-fit: cover + object-position. The <img src> on each lobby row is a data URL whose intrinsic dimensions ARE 1:1 — the crop is baked into the bitmap.

Why bother: html2canvas (the rasteriser behind the PDF export) ignores CSS object-fit and object-position, so a portrait hero with a carefully-chosen offset rendered correctly in the live lobby but came out squashed into the 1:1 slot on the PDF. Baking the crop into the image bitmap fixes both surfaces with one path.

Pure refactor + bug fix. The live lobby should look identical to before — same crop honoured for portrait sources, same scroll/hover behaviour. The PDF gets the fix as a free side effect.

In scope: web player lobby (standalone Safari + embedded WebView preview); PDF export path; cache-correctness for treatment switching.

Out of scope: Flutter Studio / filmstrip / camera peek crop math (deferred follow-up in docs/BACKLOG.md); deck active-slide rendering (uses the resolver via a different surface, untouched here); rest cards (no hero).
