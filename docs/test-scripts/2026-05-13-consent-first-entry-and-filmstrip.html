2026-05-13 · UX hygiene wave

Device QA — consent first-entry + filmstrip refresh

Scope: two coupled UX-hygiene fixes shipped in one PR against staging.

Fix 1 · Auto-open consent on first client entry. Capture defaults default to B&W + body-focus-off (PR #309). If the client's video_consent.grayscale is off, B&W playback fails. Practitioner doesn't learn this until plans break. New behaviour: ClientSessionsScreen auto-opens the consent sheet on the first-ever entry to a client (covers BOTH newly-created clients AND legacy clients whose consent was never explicitly toggled — no backfill on purpose).
Backed by new column clients.consent_explicitly_set_at TIMESTAMPTZ stamped by every call to the set_client_video_consent RPC. ClientSessionsScreen.initState checks consentExplicitlySetAt == null and opens the sheet on the next frame.

Fix 2 · Session card filmstrip refresh on session exit. Capture a new exercise inside a session, exit back to ClientSessions — the card's filmstrip should show the new hero. Pre-fix, the parent's _loadSessions() fires before the converter has stamped thumbnail_path, so the new exercise had no hero to render. Fix subscribes ClientSessionsScreen to ConversionService.onConversionUpdate and re-pulls SQLite whenever a conversion completes for any session in our list. Refresh fires on session EXIT only (not on entry — Carl signoff).

Prerequisites

  • This PR merged to staging + staging Supabase branch applied the migration 20260513065845_consent_explicitly_set_at.sql.
  • App built from staging with the SQLite v38 → v39 ADD COLUMN migration. After install, check Settings · About shows the build SHA.
  • At least one EXISTING legacy client whose consent has never been touched (older clients from before today).
  • One brand-new client created post-install for the fresh-create path.

Fix 1 · Consent auto-open

  • 1. New client → consent sheet auto-opens on first entry. From Home, tap New client (FAB). The client appears with a default New client #XXXX name. Tap into the client. Expected: the consent bottom sheet auto-opens (expanded, full content visible) immediately after the screen renders. No tap required.
  • 2. Toggle anything + save → sheet stops auto-opening. On the just-opened sheet from item 1, toggle ANY consent value (e.g. flip Black & white on or back off) and tap Save. The sheet closes. Tap back to Home, then tap back into the same client. Expected: the sheet does NOT auto-open. The detail page renders normally; consent is reachable via the existing tap path (toggle row, etc.).
  • 3. Legacy client → sheet auto-opens on first entry post-deploy. Find a client created BEFORE the migration was applied (its consent_explicitly_set_at is NULL because no backfill). Tap into it. Expected: the consent sheet auto-opens, exactly like a fresh client. The fact this triggers re-asking on every legacy client is intentional — Carl-signoff.
  • 4. No regression: already-set clients DON'T auto-open. Find a client whose consent has been explicitly toggled in this session (i.e. you already closed an item-2 save). Tap back into them — twice if needed. Expected: sheet does NOT auto-open on any entry. Same behaviour as before this PR.
  • 5. SQLite v38 → v39 ADD COLUMN survives upgrade. Before this build was installed, the app was on v38. Open Settings · About to confirm the build SHA. Tap into any client. Expected: no SQLite error, no missing-column crash. Clients list still renders; cached_clients.consent_explicitly_set_at exists (legacy rows return NULL → auto-open fires per item 3).

Fix 2 · Filmstrip refresh on session exit

  • 6. Capture a video inside a session → filmstrip updates on exit. Open any client, tap New Session. Capture one video. Exit back to ClientSessions (back chevron on Studio). Expected: the session card's filmstrip background shows the new video's Hero frame (B&W). Wait up to ~5s if the converter is still running — the filmstrip should auto-refresh the moment conversion finishes (no pull-to-refresh required).
  • 7. Capture a photo → filmstrip updates. Re-enter the same session, capture one photo. Exit. Expected: if there were no videos in the session, the filmstrip now shows the photo's line drawing. If there were already videos, the filmstrip still shows the (up to 4) videos — photo is not promoted unless there are zero videos. Either way no stale render.
  • 8. Delete an exercise → filmstrip updates. Enter the session, swipe to delete one of the exercises that's currently in the filmstrip. Exit back to ClientSessions. Expected: the filmstrip on the card no longer shows the deleted exercise's hero — the remaining heroes flex to fill the card width. No ghost cell.
