2026-05-02 · Pull-sessions fix — fresh-install hydration

Device QA — Cloud-pull sessions on fresh install

Purpose: verify that signing in on a freshly-installed device populates the session list from cloud. Before this fix, SyncService.pullAll only pulled practices, clients, credit balance, and the legacy plan→client backfill — there was no cloud→local sessions branch. Sessions only entered SQLite via on-device capture, so a reinstall (or sign-in on a second device) showed every client with zero sessions and an empty ClientSessionsScreen.

In scope: new SECURITY DEFINER RPC list_practice_plans(p_practice_id) returns each plan with its exercises + per-set rows in a single jsonb payload. ApiClient.listPracticePlans typed wrapper. SyncService._pullSessions branch added to pullAll's parallel Future.wait. LocalStorageService.upsertCloudPlanIfMissing atomic transaction inserting plan + exercises + sets only when the local row is absent (local-wins on collision).

Out of scope: remote re-download of media files (raw video / line drawing / archive). Practitioner preview plays from local files only — pulled sessions whose media doesn't exist locally render with empty thumbnails / failed playback, by design. Adding round-trip repair for missing media is a separate wave. Web-player consumption path is unaffected (it reads get_plan_full, not the new RPC).

Prerequisites

  • Branch claude/clever-euler-1d2a03 installed via ./install-device.sh.
  • Carl's iPhone 00008150-001A31D40E88401C.
  • Stream device logs alongside the test: idevicesyslog --udid 00008150-001A31D40E88401C 2>&1 | grep -E 'SyncService|listPracticePlans|pullSessions|getSessionsForUser' in a terminal.

1 · Cloud→local hydration on cold launch

The fundamental fix — sign-in fires pullAll, the new branch pulls plans from cloud, SQLite gets hydrated, the Home / clients list re-renders.

  • 1. Cold-launch the freshly-installed app. Sign in (password or magic link). Within ~3-5s of onAuthStateChange(signedIn) firing, os_log shows SyncService: pullSessions practice=<uuid> — cloud=N inserted=N skipped_existing=0 with N matching the cloud plan count for your practice (run supabase db query --linked "SELECT COUNT(*) FROM plans WHERE practice_id = '<your-practice-id>' AND deleted_at IS NULL;" to confirm the expected count).
  • 2. After the pull lands, the Home / clients list re-renders. Each client with at least one cloud session now shows a non-zero session count under their name. The "Big Boss Physio's" practice (or whichever practice you're using) renders with the same client→session distribution you had before the rebrand.
  • 3. Tap into a client with multiple sessions → ClientSessionsScreen renders the full list, ordered newest first. Each card shows the session title (Session for <DD Mon YYYY HH:MM> format), version chip, last-published time, and the dirty / clean indicator that matches the cloud state.

2 · Pulled sessions are openable + playable (within local-files limits)

Open a pulled session and verify the Studio shell hydrates. Media playback may be empty (cloud media wasn't pulled — out of scope). Per-set rows + exercise metadata MUST be intact.

  • 4. Tap a pulled session card. SessionShellScreen opens. Studio mode renders every exercise card from the pulled list — exercise count matches the cloud plan's exercise count. Each card shows its name (or position number if unnamed), notes, and the per-set row count beneath the title. Rest periods render as compact rest bars between exercise cards.
  • 5. Tap an exercise card to open the inline tray. Reps, hold seconds, breather, and (if set) weight match what was originally published. Per-set count matches what's in cloud (run supabase db query --linked "SELECT exercise_id, position, reps, hold_seconds FROM exercise_sets WHERE exercise_id = '<ex-id>' ORDER BY position;" against a sample exercise).
  • 6. Open Preview on a pulled session. The plan-preview deck renders (one card per exercise + circuit unrolling). Media tiles either show local media (impossible on a fresh install — file's not there) or fall through to a placeholder without crashing. The progress-pill matrix at the top renders with the right pill count + grouping.

3 · Local-wins guard — no clobber on collision

The new branch only INSERTS missing rows. A second pull on the same data should not re-insert; an in-flight local edit should not be overwritten.

  • 7. After the first sign-in pull lands (item 1), tap Force sync now in Settings → Diagnostics. os_log shows SyncService: pullSessions practice=<uuid> — cloud=N inserted=0 skipped_existing=N. Every cloud plan was already local; nothing was re-inserted. The clients list does NOT flicker or rerender.
  • 8. Edit a pulled session offline (airplane mode ON, rename the title). Force-sync — the local rename is preserved (the queued rename_session op is in pending_ops); the cloud title hasn't changed yet. Re-enable network. Queue flushes; cloud and local now agree. No double-rename, no flicker back to old title.

4 · Practice-switch hydrates the new practice

pullAll fires per-practice. Switching practices should pull THAT practice's sessions.

  • 9. From Home, tap the practice chip top-left → switch to a second practice you're a member of (e.g. CHM Physio Inc). os_log shows SyncService: pullSessions practice=<new-uuid> — cloud=M inserted=M skipped_existing=0 for the second practice. Clients list re-renders with that practice's clients + non-zero session counts.
  • 10. Switch back to the first practice. pullSessions reports inserted=0 skipped_existing=N — every plan we pulled the first time is still local. Switching practices does NOT clobber the other practice's sessions out of the local DB.

5 · No regressions on the capture path

A sanity pass to make sure the new pull doesn't disturb the on-device capture flow.

  • 11. From Home, create a new client → start a new session → capture one photo or short video → swipe to Studio → confirm exercise card renders. Conversion finishes within the usual window. Force-sync; no unexpected log entries about the new session being "skipped_existing" before publish (it wasn't on the cloud yet).
  • 12. Publish the new session. After publish completes, run Force sync now. os_log pullSessions branch reports skipped_existing=N+1 (the freshly-published plan is now both in cloud and local; pull respects local-wins and leaves it alone).
Pass-notes feed the next wave's polish list. Failures will spawn a targeted follow-up. Out-of-scope items observed during this pass (missing media on pulled sessions, etc.) should be flagged in the fail note for triage rather than re-tested here.
