2026-05-13 · Lobby share-as-PNG modal exclusion (D10)

Device QA — Share Modal Excluded From Lobby Snapshot

Purpose: verify that when the practitioner taps the lobby Share button, the rasterised PNG contains only the plan content (pill matrix, hero filmstrip, exercise rows) and never the "Your plan, ready to share" modal overlay that's on screen during snapshot generation.

Root cause: the share modal (#lobby-export-modal) is position:fixed; inset:0 appended to <body>, made visible with a "Generating preview..." spinner at click-time. html2canvas walks the document tree and bakes any fixed-position element overlapping the snapshot bounding box into the rasterised output — so the modal showed up inside the exported PNG.

Fix: web-player/lobby.js adds ignoreElements to the html2canvas options object. The callback skips both #lobby-export-modal and the (also potentially fixed-position) #lobby-self-grant-modal. Modal stays in the live DOM (user still sees it); excluded only from the rasterised bitmap.

In scope: lobby share path on Mobile Safari (staging public web player) and embedded preview (Flutter WKWebView via workflow Preview tab). PNG content. Live-DOM modal behaviour.

Out of scope: CORS-taint preload (unchanged), B&W bake (Bundle 1 of audit, unchanged), the Download anchor / share-sheet routing (unchanged), self-grant modal copy.

Prerequisites

  • Build from fix/lobby-share-modal-exclude branch installed on Carl's iPhone via ./install-device.sh, OR Vercel staging deploy of the same branch reachable at staging.session.homefit.studio.
  • At least one published staging plan with 3+ exercises (so the snapshot is visually rich — easier to spot whether a modal overlay is baked into the image).

  • 1. Tap Share from the lobby. The "Your plan, ready to share" modal appears immediately with a "Generating preview..." spinner. This is intentional — the modal must still appear in the live DOM.
  • 2. Within 2-3s the spinner swaps to the rendered PNG preview. The PNG itself must NOT contain the modal overlay — no "Your plan, ready to share" title, no help text, no × close button baked into the image. Only plan content (pill matrix top, exercise rows, "powered by homefit" footer).
  • 3. iOS share sheet auto-opens with the PNG attached. The thumbnail preview in the share sheet shows the clean plan snapshot — no modal overlay in it either.
  • 4. Dismiss share sheet → in the still-visible modal, tap Download PNG. Safari downloads file homefit-lobby-<timestamp>.png. Open it from Files app and confirm the downloaded image also has NO modal overlay.

  • 9. From app/: dart run tool/sync_web_player_bundle.dart --check exits 0 with "bundle in sync" output. (CR-OK — no device install needed.)
Root cause: web-player/lobby.js's triggerLobbyShare calls showExportModalLoading() at click-time, which inserts #lobby-export-modal (position:fixed inset:0) into the DOM and makes it visible with a spinner. html2canvas, when rasterising $lobby, walks the document tree and includes any fixed-position element overlapping the bounding box — even when it's a sibling of the snapshot root rather than a descendant. Fix adds ignoreElements: (el) => el.id === 'lobby-export-modal' || el.id === 'lobby-self-grant-modal' to the html2canvas options object. Modal stays in the live DOM (visible to the user during render); excluded only from the rasterised PNG output.
