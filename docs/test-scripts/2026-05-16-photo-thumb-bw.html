Device QA — Photo _thumb_bw.jpg baked bytes

Bug. Photos in B&W treatment rendered colour in the embedded preview's PDF export. Root cause: photos used the colour JPG _thumb_color.jpg as the B&W poster and relied on a CSS filter: grayscale(1) contrast(1.05) composited at render time. Videos use baked greyscale bytes in _thumb.jpg via the segmented body-focus pipeline, so the same logical treatment ended up with two different render mechanisms. PR #368 patched the canvas-resolver edge case; this PR fixes the upstream bytes.

Fix. Bake a sibling _thumb_bw.jpg at photo conversion time. The OpenCV isolate now applies BGR2GRAY followed by convertScaleAbs(alpha=1.05, beta=-6.375) — visually equivalent to the CSS filter the lobby would otherwise composite. The Supabase get_plan_full RPC surfaces the new variant as thumbnail_url_bw via an existence check against the public media bucket; web-player/exercise_hero.js prefers it for B&W when present and suppresses the CSS filter so already-grey bitmaps don't take the contrast curve twice. get_plan_full was sourced via pg_get_functiondef — every existing field is preserved verbatim.

Legacy backfill. SQLite v40 → v41 stamps thumbnails_dirty = 1 on every existing photo row. ConversionService.backfillMissingVariants (already wired to fire on Studio mode session-open) re-runs the OpenCV isolate when _thumb_bw.jpg is missing on disk; the next publish then re-uploads the freshly-baked file via the existing thumbnailsDirty override path. So legacy plans pick up the bake within one publish cycle of the practitioner installing this update.

R-10 parity. Both player surfaces touched: the web player (web-player/exercise_hero.js) and the Flutter-bundled mirror (app/assets/web-player/exercise_hero.js) ship byte-identical. diff -q verified.
