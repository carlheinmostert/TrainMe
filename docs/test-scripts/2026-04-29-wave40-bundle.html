B · Portal hygiene (P1–P8)

  • 8. P1 · Top-bar nav links retired. Header is logo + right cluster only.
    1. Hard-refresh https://manage.homefit.studio/dashboard (Cmd-Shift-R). Sign in if needed.
    2. Look at the top bar. The middle/right area NO LONGER shows the row of links Clients · Credits · Network · Audit · Members · Account.
    3. What remains: the homefit.studio logo + wordmark on the left, and on the right the practice-switcher chip + email chip cluster (covered by items 9 & 10).
    4. Navigate via the dashboard tiles instead. Each tile is the destination — tiles ARE the menu.
    5. Repeat the visual check on /clients, /audit, /credits, /network, /members, /account: same minimal header on every signed-in page.
  • 9. P2 · Signed-in email chip on the right with dropdown.
    1. On /dashboard, look at the far-right of the top bar. There's a chip with your signed-in email (e.g. carlhein@me.com) and a small caret ⇓ after it.
    2. Hover the chip — border picks up coral. The full address is also exposed via the title attr on hover.
    3. Click the chip. A dark popover opens BELOW the chip with two items: Account settings + Sign out.
    4. Click outside the popover — it closes. Press Escape with it open — it closes.
    5. Click Account settings — navigates to /account?practice=… (active practice id preserved).
    6. Open the popover again, click Sign out — cookie clears, redirected to /.
  • 10. P3 · Practice switcher chip sits to the LEFT of the email chip.
    1. On /dashboard, the right-cluster reads (right-to-left): [email chip] · [practice chip]. The practice chip shows the active practice name (e.g. BigBoss) and an icon (⇔-style swap arrow).
    2. If you only have ONE practice, the chip renders as static muted text (the active name) without a clickable border — nothing to switch to.
    3. If you belong to >1 practice (use the test rig with multi-practice auth), click the chip. Same Milestone N popover from the dashboard In practice: … line opens: list of practices with the active one anchored at the top + check-mark.
    4. Click another practice in the popover — URL gets ?practice=<newId>, page reloads in that context (header chip + tile data both reflect the switch).
    5. Verify cluster order on every signed-in page: dashboard, clients, audit, credits, network, members, account. Practice chip ALWAYS left, email chip ALWAYS right.
  • 11. P4 · Dashboard audit card shows the 5 most-recent events (real data, not empty).
    1. On /dashboard, find the Audit tile. It now spans wider than the other tiles (col-span-2 on small breakpoints).
    2. Headline reads {N} events where N ≤ 5 (matches the rows below).
    3. Below the headline: a divider list of up to 5 rows. Each row has a small kind chip (sage / coral / red / grey from auditChipTone), an actor label (email or "Client" for anon plan.opened), and a relative timestamp.
    4. Hover the card — border lights coral; the chevron in the upper-right (subtitle "See full log") translates 0.5px right.
    5. Click anywhere on the card — navigates to /audit?practice=…. Practice id propagates via the qs.
    6. Empty practice (a fresh signup with no events): card headline reads No events yet + subtitle Publish from mobile to fill this. Same shape; just no list inside.
  • 12. P4 · Dashboard audit card timestamps render in browser-local TZ via <ClientTime/>.
    1. On /dashboard, look at the timestamps in the audit card's row list. Format is "DD Mon YYYY, HH:MM" — same format as the dedicated /audit page.
    2. If the Mac TZ is Africa/Johannesburg (SAST, UTC+2), timestamps match the iPhone wall clock for the most-recent publish.
    3. Switch System Settings TZ to America/Los_Angeles. Hard-reload the dashboard. Same rows now read in Pacific time (10h behind SAST). Numbers visibly different.
    4. Switch the Mac TZ back to Africa/Johannesburg.
  • 13. P5 · Audit kind legend wraps inline (1-2 lines instead of 5).
    1. On /audit, scroll to the Kind filter section in the filter bar.
    2. The chip rows used to occupy ~5 vertical lines (one per group: Plan / burn, Credits in, Destructive, Neutral). After Wave 40 P5 they collapse into 1-2 wrap lines depending on viewport width.
    3. Each group label still reads as a quiet uppercase <small> left of its cluster of chips. A thin vertical separator divides one group from the next.
    4. Click any chip — it toggles selected state (filled background + coral/sage/red ring). URL updates with ?kinds=…; page reloads.
    5. Resize the browser window narrow (~600px). Chips re-wrap; group order preserved.
  • 14. P6 · Avatar glyph on every /clients list row.
    1. Open https://manage.homefit.studio/clients?practice=….
    2. Each client card shows a coral-tinted disc with the client's initials (1-2 letters, uppercase) immediately to the left of the client name. Single-name clients = first letter; two-or-more-word names = first letter of first word + first letter of last word.
    3. Cards reflow correctly across breakpoints — avatar + name on a single row at desktop, still readable when the grid drops to one column.
    4. Disc has a subtle coral border (40% alpha brand) + brand-tint background. Letters are coral, semibold, font-heading.
    5. No regression: the existing "{N} sessions · shared {ago}" line + (owner-only) "Practitioner: {email}" line still render below the name.
    6. Hover-only delete button still appears bottom-right of each card on hover/focus and stays clickable (doesn't collide with the avatar).
  • 15. P7 · Clicking a client on /clients drills into /clients/[id] (not /dashboard).
    1. From /clients?practice=…, click any client card.
    2. Browser navigates to /clients/<uuid>. Middleware strips ?practice= from the URL after pinning the cookie. The detail page renders normally with the client's consent panel + sessions list.
    3. CRITICAL pre-fix regression: the page used to redirect to /dashboard because middleware stripped the qs and the detail page didn't fall back to the hf_active_practice cookie. After Wave 40 P7, the cookie fallback mirrors /clients, /audit, /credits.
    4. Open a client tile in a new tab via Cmd-click. Same flow — tab opens at /clients/<uuid> with the right client.
    5. Hard-refresh the detail page (Cmd-Shift-R). Stays on the detail page (cookie persists). No bounce to /dashboard.
  • 16. P8 · Session-icon parity inside /clients/[id].
    1. Pick a client with at least one published session. Open /clients/<uuid>.
    2. Scroll to the Sessions table. Each row carries a small avatar disc (initials of the client) on the LEFT, with a coral-circle badge in the bottom-right showing the exercise count (1, 2, 3, …).
    3. Badge has a 2px --bg-coloured ring around it so it visually pops off the avatar disc — mirrors the mobile SessionCard treatment.
    4. On narrow viewports (sm:hidden card stack), the avatar still leads the row above the title + relative timestamps.
    5. Sessions with 0 exercises (rare edge case) hide the badge entirely. Avatar disc still renders.
    6. Until list_practice_clients threads avatar_path through the RPC, the disc shows initials only. The component is wired to swap in the body-focus blurred avatar JPG the moment the column lands.
