Baseline recovery phase 2 — auth.users trigger

Why this matters. Same root cause as PR #354: the 2026-05-11 baseline migration was generated via pg_dump --schema=public, which silently DROPPED any trigger whose target table lives outside the public schema. The function public.claim_pending_practice_memberships() made it into the baseline, but the AFTER INSERT ON auth.users trigger that invokes it did not. Result: on every Supabase Branching-cloned DB (staging + every per-PR preview), an owner can invite a practitioner by email, the row lands in pending_practice_members, but when the invitee signs up nothing fires — the pending row never drains and the invitee lands without their expected practice membership. Production has the trigger (it was applied directly when milestone U shipped) so this only bites on staging + previews.

The migration. supabase/migrations/20260515140953_baseline_recovery_phase2_auth_trigger.sql restores the trigger verbatim from supabase/archive/schema_milestone_u_add_member_by_email.sql:450–455. Pre-flight check confirms the trigger function exists in the baseline before creating the trigger. Idempotent: DROP TRIGGER IF EXISTS then CREATE TRIGGER, wrapped in a DO block with insufficient_privilege exception handling.

Sensitive zone — auth-schema trigger. Carl review before merge.
