QA — circuit animation attempt #10 (nested CSS boxes)

Scope: client web player session.homefit.studio + iOS embedded preview (workflow Preview tab) + PDF export. Mobile Flutter player is unaffected.

The story so far: nine previous attempts (PRs #257/#258 → #260 → #317 → #322 → #337 → #342 → #353 → #360) all patched layers of an SVG-tracer architecture that depended on runtime DOM measurement, ResizeObserver, MutationObserver, getTotalLength() polling, font-load awaits, and WAAPI-vs-CSS cascade priority on position:fixed ancestors. Every fix introduced new failure modes; the most recent (PR #360) was a stopgap.

Attempt #10: the entire SVG architecture is gone. Circuit chrome is now N visually-nested <div class="lobby-circuit-box"> rings (one per cycle, capped at 5) emitted by circuitGroupHTML() in lobby.js, animated by a single CSS keyframe lobby-circuit-pulse staggered via the --box-index custom property. No JS animation. No measurement. No observers. No requestAnimationFrame chain. The whole class of races cannot recur.

Mockup: docs/design/mockups/circuit-nested-boxes.html (variant 1, outward ripple). The keyframes in the live CSS are the same shape as the mockup's v1-pulse.

Install: on Carl's iPhone via ./install-device.sh after the PR merges to staging. The web surface (session.homefit.studio) auto-deploys on Vercel from the merge. Service worker cache is bumped so Safari clients flush the stale SVG-tracer code paths.

Prerequisites

  • You need at least one published plan with a circuit (the trainer app: add 2+ exercises to a circuit, set the rounds chip to the count you want to test).
  • Test with multiple plans: one with ×1 circuit, one with ×3, one with ×6 (to verify the cap).
  • Hard-refresh the web player (Cmd+Shift+R on desktop, or kill + relaunch the iOS app for the embedded preview) so the service worker picks up the new cache name.

1 · Nested boxes render per circuit count

  • 1. ×1 circuit renders 1 box. Open the lobby for a plan whose circuit has cycles = 1. Expected: the circuit group is wrapped in exactly ONE coral-bordered rounded rectangle around the exercise rows. The header strip (CIRCUIT NAME · ×1) sits above the box. A single rectangle pulses softly (border brightens + faint coral glow every ~2.4 seconds).
  • 2. ×2 circuit renders 2 nested boxes. Open a plan whose circuit cycles = 2. Expected: two concentric rounded rectangles. The outer one wraps the inner one; the inner one wraps the rows. The two rings pulse with a small stagger (~0.4 s) so the brightening visibly ripples outward from the inner ring.
  • 3. ×3 circuit renders 3 nested boxes. Open a plan whose circuit cycles = 3. Expected: three concentric rounded rectangles; each successive outer ring has a larger radius than the one inside it. The pulse staggers visibly across the three rings — you should be able to count three distinct brightening moments per cycle, propagating outward.
  • 4. ×5 circuit renders 5 nested boxes. Open a plan whose circuit cycles = 5. Expected: five concentric rings. This is the cap (CIRCUIT_BOX_CAP = 5); the count chip still reads ×5. The whole nest fits inside the lobby column without horizontal scroll.
  • 5. ×6+ circuit still renders 5 boxes (cap respected). Open a plan whose circuit cycles = 6 (or higher). Expected: only 5 nested rings are drawn — the same as ×5 above. The header chip should read the TRUTHFUL count (×6 / ×7 / etc.) so the practitioner's actual rounds aren't hidden; only the chrome caps. No horizontal scroll, no overflow visual.

2 · Animation behaviour

  • 6. Pulse is visible — not subtle, not over-the-top. Watch a ×3 circuit on the lobby for ~10 seconds. Expected: each ring visibly brightens to full coral + emits a soft glow (~24px box-shadow), then settles back. The peak should be obvious from across the room but not strobe-level distracting. If you can't see any movement, the keyframe is inert (regression of the 6-variant mockup's "too subtle" problem). If it's pulsing so hard it's uncomfortable, we tuned too high.
  • 7. Outward ripple direction. On ×3 / ×5, the brightening should propagate INNER → OUTER (the inner ring lights up first, then the middle, then the outermost). If it ripples the wrong way or the rings light up in unison, the --box-index × animation-delay mapping regressed.
  • 8. Animation runs steadily — no pause, no stutter, no main-thread peg. Scroll the lobby up and down a few times, switch slides in the deck, then come back to the lobby. The circuit boxes should still be pulsing smoothly. The page should remain interactive (tap a row, scroll). Any stutter or page-freeze on circuit plans means a JS path is still touching the chrome.
  • 9. prefers-reduced-motion kills the animation. In Settings → Accessibility → Motion → Reduce Motion, enable the toggle, then re-open the lobby. Expected: the nested boxes are visible (static coral borders, dim) but do not pulse. Disable the toggle again — the pulse should resume on the next page load. If the animation keeps running with reduce-motion on, the @media (prefers-reduced-motion: reduce) rule isn't reaching.

4 · PDF export parity

  • 13. PDF preview shows the nested boxes (static). Tap Share, wait for the PDF preview modal. Expected: on the rendered page, each circuit group is wrapped in N visible coral-bordered rings (same count as the live lobby). The chrome is rasterised as a SETTLED frame — no animation, no half-pulsed state, no missing borders. The CSS overrides border-color to a bright coral so the rings read clearly in print.
  • 14. PDF — no clipped right edge on the outermost ring. Inspect the rendered page carefully. The outermost ring's right border must not be cropped by the page edge. (The export CSS forces overflow:visible on every level the chrome nests inside.) If you see a half-rendered ring, the export pipeline is clipping a parent.
  • 15. PDF — rows render inside the innermost ring, not below it. The in-circuit exercise rows must sit visually inside the inner ring's padded area. If the inner-ring border draws around an EMPTY rectangle with the rows below it, the row clone path lost its parent in the DOM clone.
  • 16. PDF download succeeds. Tap Download in the PDF preview modal. Expected: the PDF saves to your device's downloads (or iOS Files / shares via the native sheet). Open the saved PDF — it should look identical to the in-app preview (nested rings + rows inside the inner one).

  • 17. No DOM measurement — rotate the device, resize the window. On the lobby with a circuit plan, rotate the iPhone (or resize the browser tab on desktop). Expected: the nested boxes re-flow with the lobby column width but the animation does NOT pause, glitch, or restart. No ResizeObserver is wired to the chrome, so geometry changes are pure CSS reflow.
  • 18. No observer feedback loop — switch slides rapidly. In the deck, tap forward/back chevrons rapidly for ~5 seconds. Return to the lobby. Expected: the page stays interactive (scroll works, taps land, animation continues). The ninth-attempt symptom of "main thread pegged" from observer recursion cannot recur because there is no observer.
  • 19. Active-row coral highlight still works on in-circuit rows. Tap an exercise row inside a circuit. The row should get the existing coral "active row" treatment (border + tint). The nested boxes should not interfere with row-level styling. If active-row state is invisible, the new ring borders are over-painting the row chrome.

Reload the lobby. Expected: no console errors. If you see renderCircuitLanes is not defined or any reference to the deleted SVG architecture, there's a dead caller still pointing at the old code.
