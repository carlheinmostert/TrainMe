2026-04-21 · Wave 4 Phase 2 — postMessage bridge + URL scheme

Device QA — Wave 4 Phase 2

Purpose: verify the unified player now runs through the production-grade bridge: Flutter posts plan state to the WebView via postMessage, the WebView resolves local media via WKURLSchemeHandler with HTTP Range support, and the Phase 1 Dart shelf loopback is fully retired. Native bits (Taptic haptics, iOS audio session) stay on the Flutter side and bridge through message handlers.

In scope: plan-state postMessage, homefit-local:// URL scheme handler, Range request streaming, haptic bridge, audio session bridge, fallback path when WebView load fails.

Out of scope: web-player UI parity with old Dart preview (covered in Phase 1 script), raw-archive uploads, publish flow, Wave 5 Members.

Prerequisites

  • Wave 4 Phase 1 unified-player prototype PR has merged and ./install-device.sh has installed the new build.
  • Plan with at least one video exercise on Line + B&W + Original available in a client's sessions list.
  • Airplane mode available on the test device so offline paths can be exercised.

1 · Plan-state bridge (Flutter → WebView postMessage)

Flutter serialises the plan as JSON and posts it to the WebView once on load. No more per-request HTTP GET against a local Dart server.

  • 1. Open a plan preview from Studio — the WebView renders the pill matrix, exercise name, and video controls within 1.5s of tap. No visible blank-screen gap while the bridge hands off state.
  • 2. Edit an exercise's reps in Studio → return to preview — the WebView shows the updated reps immediately on the first slide (state repost on resume).
  • 3. Airplane mode ON → open the same plan preview — the WebView still loads + renders from cached state (no network round-trip to Supabase).

2 · WKURLSchemeHandler + Range request streaming

Local files resolve via a custom homefit-local://{exerciseId}/{treatment}.mp4 scheme handled in Swift. The handler must honour HTTP Range headers so WKWebView can seek into large videos without full downloads.

  • 4. Inspect a playing video slide's source via Safari Web Inspector — the <video> src is a homefit-local:// URL, not file:// and not http://localhost:....
  • 5. Scrub the video progress bar forward and back — playback seeks within ~300ms, no full reload, no broken-frame flash. Range requests working.
  • 6. Switch treatments via the preferred-treatment UI on a plan that has Line + B&W — video swaps cleanly, no stall, no blank frame, no audio cut-out.
  • 7. Navigate through 5+ exercise slides — each video starts playback within 500ms of becoming active. No cumulative slowdown across slides.

3 · Retirement of shelf loopback

Phase 1's Dart shelf HTTP server on localhost is gone. No process is listening on the old port; no http://localhost: URLs appear in the WebView.

  • 8. Run lsof -iTCP -P | grep LISTEN on the device via Xcode's device console (or check via Settings → connections) — no Dart-side port listener bound by the Flutter app while a preview is open.
  • 9. Grep the installed Flutter bundle (or inspect sources via git log) — the shelf dependency is removed from pubspec.yaml; no residual server startup code in main.dart or any service.

4 · Haptic + audio-session bridge (native-side, from WebView)

Per-second record haptics, prep-countdown flashes, and iOS silent-switch respect all still fire — even though the player now runs in a WebView. The WebView posts messages back to Dart, which calls native code.

  • 10. Start a workout from preview — the 5-second prep countdown fires a subtle haptic tick each second. No silent countdown. (Audio-session category routes through AVAudioSession so haptic engine is available.)
  • 11. Flip the iPhone silent switch ON → play a video with audio — audio is suppressed (respects silent). Flip OFF → audio returns on the next play. No "mute gets stuck" regressions.
  • 12. Background the app mid-exercise and return — video resumes from where it paused; haptic clock resumes on next prep. No bridge teardown leaking state.
