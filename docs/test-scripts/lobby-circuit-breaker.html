Device QA · iPhone · Lobby circuit-breaker diagnostic (v46)

Lobby circuit-breaker — surface the rAF / scroll runaway behind the WebView freeze

PR #249's smooth-scroll fix did NOT resolve the iOS WKWebView lobby freeze. After ~5–8s of sustained scrolling the WebView still "goes black", the heartbeat console.log dies, no JS errors, no network. A profiler attempt OOM'd. This wave ships a per-callsite circuit breaker that aborts + logs whenever an instrumented rAF / setTimeout / scroll-coupled callback exceeds 60 calls / 1000ms.

9 sites instrumented in web-player/lobby.js: recomputeActiveRow, setActiveRow, lazyKickVideosNear, scrollRaf, treatmentRaf, popoverOpenRaf, selfGrantRaf, popoverCloseTimeout, heroErrorRetry.

Heartbeat: every 2 seconds the breaker logs [CB-hb] {name1: count, name2: count} for any instrumented name with > 0 calls in the trailing 1s. Visible in Safari Web Inspector even when nothing trips. Wired exactly once inside showLobby().

Trip behaviour: first call past 60/sec logs [CB] <name> runaway: N calls in last 1000ms — aborting and bails out. Subsequent dropped calls log once with [CB] <name> suppressed (already tripped). Auto-rearm clears the trip flag after 5s.

SW cache bumped: homefit-player-v45-lobby-scroll-freeze-hotfix → homefit-player-v46-lobby-circuit-breaker. Build chip should read web-player v46-lobby-circuit-breaker · cache v46-lobby-circuit-breaker once the new SW takes over.

Prerequisites

  • Pull / hard-refresh session.homefit.studio in mobile Safari (kill app + relaunch) so the new SW activates. Confirm the build-marker chip reads v46-lobby-circuit-breaker on both bundle + cache (divergence = stale SW).
  • Connect iPhone to Mac via USB. Open Safari (desktop) → Develop → [Your iPhone] → session.homefit.studio. Switch to the Console tab. This is the only way to see [CB-hb] + [CB] runaway on device.
  • For the WebView path (Workflow Preview): open the Flutter app on your iPhone, drive a session into Workflow Preview, then in desktop Safari Develop → [iPhone] choose the WebView entry under the Flutter app process.
  • Plan should have at least 8–10 rows so there's enough scroll runway to provoke the freeze.

1 · Heartbeat is alive

  • 1. Lobby loads + heartbeat appears in console. Open the published session URL (or Workflow Preview) on iPhone and watch the desktop Safari Web Inspector console. Expected: within 4 seconds of the lobby rendering, a line of the form [CB-hb] {"recomputeActiveRow":1,"setActiveRow":1,"lazyKickVideosNear":1} (or similar — exact contents depend on which sites fired during the activation animation). Subsequent [CB-hb] lines appear every ~2s. Failure mode: no [CB-hb] at all means either the heartbeat never wired (showLobby didn't run) or the SW is stale (still on v45).
  • 2. Heartbeat goes silent when nothing is happening. Let the lobby sit untouched for ~6 seconds. Expected: after the activation flurry, the heartbeat stops printing — the breaker only logs when at least one name has > 0 calls in the last second. NO baseline noise. (If the heartbeat keeps printing every 2s with non-zero counts while you're not touching the screen, something is firing in a background loop and that itself is interesting data — record what's in the summary in the note.)

2 · Normal scroll — no trip

  • 3. 10 seconds of normal scrolling produces sane counts. Scroll the lobby up and down with normal-paced finger flicks for ~10 seconds. Expected: [CB-hb] lines report non-zero counts for at least scrollRaf, recomputeActiveRow, and setActiveRow. Counts should stay under ~30 per second per name in normal scroll. lazyKickVideosNear should also tick whenever setActiveRow picks a new row. NO [CB] runaway messages during normal scroll.
  • 4. Scroll feels identical to v45. Lobby chrome and scroll behaviour shouldn't have changed — only diagnostic instrumentation was added. Expected: matrix pill tracking, active-row coral border, hero video lazy-load, smooth feel — all unchanged from v45. (If anything feels different, note it — the breaker should be invisible to UX when not tripping.)

3 · Trigger the freeze — runaway lands in console

  • 5. Sustain a scroll long enough to provoke the freeze. Hold a finger on the lobby and drag continuously up and down with NO release for ~5–8 seconds (that's the freeze pattern Carl confirmed on PR #249). Expected at the moment of (or just before) the freeze symptom: ONE OR MORE [CB] <name> runaway: N calls in last 1000ms — aborting messages in the console. The name in that message is THE bug. Possible candidates: recomputeActiveRow, setActiveRow, lazyKickVideosNear, scrollRaf. Record the EXACT name + count in the note — that's the single most important data point of this whole wave.
  • 6. WebView remains responsive (the abort breaks the freeze). When the breaker trips it short-circuits the runaway callback. Expected: the screen does NOT go black this time — or, if it briefly stutters, recovers cleanly. Active pill may stop tracking momentarily (the breaker is suppressing recomputeActiveRow / setActiveRow for 5s). Failure mode: screen still goes fully black & heartbeat dies after a runaway message — means the abort isn't on the right callsite (the freeze is downstream of every instrumented site, or it's something we haven't instrumented).
  • 7. Auto-rearm fires after 5 seconds. After the runaway message lands, leave the screen alone. Expected: ~5 seconds later the heartbeat resumes, with the previously-tripped name no longer suppressed. (You may see a single [CB] <name> suppressed (already tripped) line during the cool-off if anything tried to call the tripped path.)

4 · Non-scroll callsites are silent in normal use

  • 8. Open / close the gear — popover counters tick, none trip. Tap the gear icon in the sticky CTA bar to open the lobby settings popover, then tap outside to close it. Expected: next [CB-hb] shows popoverOpenRaf + popoverCloseTimeout with counts of 1 each. NO runaway. If you toggle rapidly, counts climb but stay well under 60 unless you're spamming.
  • 9. Switch treatment — treatmentRaf ticks once. Open the gear and tap a different (unlocked) treatment pill (Line / B&W / Colour). Expected: next [CB-hb] shows treatmentRaf with count 1. Heroes re-render. NO runaway. Active treatment changes globally as before.
  • 10. Self-grant flow — selfGrantRaf ticks once. Tap a LOCKED treatment pill, then tap Allow in the modal. Expected: next [CB-hb] shows selfGrantRaf with count 1 after the consent grant lands. Plan re-fetches; heroes update. NO runaway. (Skip if you don't have a session whose practitioner-set consent excludes one of B&W / Colour.)

5 · Regression sanity

  • 11. Start Workout still hands off cleanly. Tap the coral Start Workout CTA. Expected: lobby dismisses, deck takes over, prep countdown fires, first exercise plays. The breaker continues to count any deferred lobby callbacks (none should fire post-handoff) but produces no further heartbeats once the lobby is gone (the heartbeat itself uses a setInterval that survives dismissal — that's by design so we still see late stragglers).
  • 12. Build chip on the lobby reads v46. Cosmetic check before you finish — the top-left chip in the lobby reads web-player v46-lobby-circuit-breaker · cache v46-lobby-circuit-breaker on both bundle + cache values. Divergence = stale SW didn't activate; reload after killing Safari from the app switcher.
