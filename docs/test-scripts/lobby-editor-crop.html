Wave Lobby · PR 2/4 · practitioner authoring UI

Editor QA — 1:1 hero-crop drag overlay on the Preview tab

Scope: the practitioner-facing UI to author the 1:1 lobby crop. Drops a coral-bordered viewport overlay on top of the displayed video / photo in the editor sheet's Preview (Hero) tab. The practitioner drags it along the source's free axis (X for landscape, Y for portrait); on release the value lands in exercises.hero_crop_offset. PR 1 already shipped the schema + RPC plumbing + model field + sync mappers; this PR wires only the editor UI.

Behaviour contract: null hero_crop_offset still means “centred crop (0.5)” for any consumer — this PR does NOT touch the consumer surfaces (Studio cards, Home, ClientSessions, Camera peek, web-player lobby). Those land in PR 3 (mobile thumbnail consumers) and PR 4 (web lobby). The only consumer that DOES read the live offset in this PR is the editor sheet's own header thumbnail, which mirrors the practitioner's drag in lock-step so they get visual confirmation as they pan.

Surfaces touched: app/lib/widgets/hero_crop_viewport.dart (new), app/lib/screens/studio_mode_screen.dart (overlay wired into the [MediaViewerBody] Stack, gated on embeddedInSheet; _persistHeroCrop mirrors _persistHero's debounced-write pattern), app/lib/widgets/mini_preview.dart (new cropOffset param maps to an [Alignment] on the static-Hero branch; default null preserves today's BoxFit.cover centred render), app/lib/widgets/exercise_editor_sheet.dart (header MiniPreview now passes cropOffset).

Prerequisites

  • PR 1 (schema) merged + Carl applied supabase/schema_wave_hero_crop.sql. Mobile installed at SQLite v38.
  • Re-install via ./install-device.sh to pick up this PR.
  • At least one published plan with one landscape video, one portrait video, and one photo on the device. Carl's existing test plans should cover this; if not, capture one of each before walking the script.

1 · Overlay appears on the Preview tab

  • 1. Overlay paints on a video exercise. Open Studio → tap a video exercise card → the editor sheet opens. Swipe to the Preview tab. The video plays. Expect: a coral-bordered 2px square (1:1) viewport sits over the video, with the area outside the box dimmed to a translucent sage tint. Inside the box stays clean (no dim, no border noise).
  • 2. Default value renders centred. On a freshly-captured exercise that the practitioner has never touched, the viewport sits at the centre of the source's free axis (i.e. the box is centred horizontally for a landscape source, centred vertically for a portrait source). The reading on the wire is still NULL — the centre default is purely a render fall-through, not a write.

2 · Drag axis follows source orientation

  • 3. Landscape source — horizontal drag only. Open the Preview tab on a landscape (16:9) video. Drag the viewport HORIZONTALLY across the displayed media: the box slides left/right smoothly tracking your finger; the dim re-paints in lock-step. Drag VERTICALLY: the box does not move; the gesture is rejected. (You may notice the underlying treatment-cycle vertical-flick fires instead — that's expected, the viewport doesn't claim vertical drags on a landscape source.)
  • 4. Portrait source — vertical drag only. Open the Preview tab on a portrait (9:16) video. Drag the viewport VERTICALLY: the box slides up/down smoothly. Drag HORIZONTALLY: the box does not move. Horizontal swipes in the Preview body should still page the editor sheet's tabs (left = next tab, right = previous). The Preview tab's chevron-nav for switching between exercises should also still work.
  • 5. Drag clamps at the edges. Drag past the source's left edge / right edge (landscape) or top / bottom (portrait): the box stops at the edge of the displayed media — it never goes past. The internal value clamps to 0.0 / 1.0.

3 · Persistence + live header mirror

  • 6. Header thumbnail updates live during drag. While dragging the viewport on the Preview tab, glance up at the editor sheet's header MiniPreview (top-left, 168×110px). It re-renders in lock-step: as you drag right on a landscape source, the header thumbnail slides its visible window right too; same for vertical on portrait. (The thumbnail itself is a static Hero-frame JPG; what's moving is the BoxFit.cover alignment as the offset bubbles up via onExerciseUpdate.)
  • 7. Value persists across sheet close + re-open. Drag the viewport to a non-default position. Release. Drag the sheet down to dismiss (or tap-outside). Tap the same exercise card again to re-open the editor. Swipe to the Preview tab. Expect: the viewport is at the position you released it at, NOT back at centre. Quickly close + reopen multiple times to confirm the debounce isn't losing the final value.
  • 8. Crop write does NOT touch the Hero frame moment. Open the Preview tab. Note the current focus_frame_offset_ms visually — pause playback at the trim panel's Hero star handle and remember the displayed frame. Drag the crop viewport to a new position. Release. The video should NOT have re-seeked to a different frame, and the trim panel's Hero star handle should NOT have moved. (i.e. the crop and the focus-frame are independent fields.)

4 · Reset + photo + round-trip

  • 9. Double-tap inside the viewport resets to centre. Drag the viewport to an off-centre position. Release. Now double-tap inside the viewport's box. The box snaps back to centre. The header MiniPreview also returns to its centred render.
  • 10. Photo exercise behaves the same. Open a photo (not video) exercise. Editor sheet → Preview tab. The viewport renders over the photo. Drag along the free axis (depends on whether the photo is landscape / portrait). Release. Re-open. Position is preserved. The header MiniPreview mirrors live during drag. Caveat: a legacy photo with no aspect_ratio stored will render the viewport as canvas-sized (drag disabled) — that's a graceful degrade, not a fail.
  • 11. Round-trip from another device. On the device, drag a video exercise's crop to ~0.25 (a noticeably-leftward / upward position) — release. From the live Supabase SQL editor, confirm: SELECT hero_crop_offset FROM public.exercises WHERE id = '<exercise-id>' returns ~0.25 (after a publish, since SQLite writes are local-only until publish). On the same device, manually update the cloud value: UPDATE public.exercises SET hero_crop_offset = 0.75 WHERE id = '<exercise-id>'. Force a sync (kill + reopen the app). Open the same exercise's Preview tab: viewport renders at ~0.75.

5 · Out of scope (verify NOT regressed)

  • 12. Studio cards / Home / ClientSessions / Camera peek render unchanged. Walk Home → ClientSessions → Studio → Camera. All thumbnail surfaces (filmstrip, list rows, peek box) still render exactly as before this PR — PR 3 hasn't shipped, so no consumer reads hero_crop_offset yet. None of these surfaces should reflect any practitioner crop you set above.
  • 13. Web player still loads existing plans pixel-stable. Open a published plan link in Safari (session.homefit.studio/p/<plan-id>). Lobby placeholder, exercise cards, prep overlay, video poster, rep stack — all render identically. PR 4 (the lobby itself) is parallel and isn't shipping in this PR.
  • 14. Full-screen workflow Preview route is unchanged. From Studio, tap the Preview button on the workflow toolbar (the route-pushed full-screen viewer, NOT the editor sheet's Preview tab). Swipe through exercises. The crop overlay should NOT render here — the overlay is gated on embeddedInSheet so the full-screen path is unchanged. PageView pagination still works on a landscape source.
  • 15. Trim panel + Hero star handle still work. On the Preview tab with the crop overlay visible, the trim panel underneath still hosts the begin / Hero ★ / end handles. Drag each handle: video pauses + scrubs as before, value persists, regenerated thumbnail flows up to the header. (i.e. the new overlay didn't accidentally eat the trim panel's gesture surface.)
  • 16. Tap-to-pause still works through the overlay. Open Preview tab on a video exercise. Tap (no movement) inside the viewport's clean area. The video pauses. Tap again. It plays. (Translucent hit-test on the overlay so the underlying tap-to-pause GestureDetector still receives the event.)
