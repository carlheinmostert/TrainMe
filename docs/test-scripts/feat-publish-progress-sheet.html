Device QA — Publish progress sheet + atomic publish

Purpose: verify the third (and largest) PR of the 2026-05-15 publish-flow refactor. The old toast-driven publish UX is replaced with a bottom sheet rendering five phase rows (Preparing · Reserving credit · Uploading treatments · Saving plan · Finalising). The sheet is non-blocking — swipe-down keeps publish running and pops a coral chip into the workflow toolbar. Every raw-archive variant upload is now required: any failure throws PublishFailedException, refunds the credit (with one retry on transient failure), and lands the sheet on its failure state with a Show which files → tap-through to the existing UploadDiagnosticSheet (PR #345).

What's gone: the legacy “Some optional treatment files are still processing…” toast is retired — publish is now atomic, so this state cannot occur. The “Published ✓” toast is replaced by the sheet's 1-second sage “All set” beat + auto-dismiss, with a sage “Plan published” chip backstop in the toolbar for ~3s afterwards.

Spec: the four wireframe variants A/B/C/D in docs/design/mockups/publish-progress-sheet.html. Sensitive zone (publish flow); Carl's autonomous-drive directive applies but a pre-merge eye on upload_service.dart's reordered Step 7.5 + throw site is the load-bearing review.

Order matters: walk items 1 → 13 in order. Items 4 and 5 require flaky network (Airplane mode mid-upload) so do them on Wi-Fi where toggling air is fast.

Prerequisites

  • Build from feat/publish-progress-sheet-atomic installed on Carl's iPhone via ./install-device.sh.
  • One fully-consented test client (B&W + Original ON; can't fail on consent gates while testing the upload path).
  • A fresh session ready to record: 3+ video captures so the “Uploading treatments” phase has multiple files to tick through.
  • Initial credit balance visible on the home credits chip — you'll compare before/after.
