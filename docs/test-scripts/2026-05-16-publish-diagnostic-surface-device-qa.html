2026-05-16 · Publish diagnostic surface + atomic media uploads

Device QA — Publish failures always surface diagnostic data

Purpose: verify two complementary fixes that close the gap PR-C left between atomic raw-archive uploads and the rest of the publish flow.

Fix A (fallback "Show error details →" tap-target): when publish fails with no per-file failure list — network drops, RLS rejections, RPC errors, credit-consume blips, savePlan errors — the progress sheet now renders a new "Show error details →" link that opens UploadErrorDetailsSheet with the phase that failed, the exception type, the user message, the technical detail, and a Copy button.

Fix B (atomic media-bucket uploads): every _api.uploadMedia call in the line-drawing + thumbnail + line-variant-thumbnail upload loops is wrapped in per-file try/catch. Failures collect into a mediaFailures list; after the loop, if non-empty we throw PublishFailedException with the records — same shape the raw-archive pass uses since PR-C. The progress sheet's existing "Show which files →" link now fires for media-bucket failures too, opening the existing UploadDiagnosticSheet.

Stacks on: PR-C atomic publish flow (PR 7548b3b) + PR #357 useRootNavigator: true on the diagnostic sheet.

Out of scope: web-player changes (R-10 doesn't apply — this is mobile chrome only), Supabase RPC changes, raw-archive flow (already atomic since PR-C).

Sensitive zone: publish flow (app/lib/services/upload_service.dart). Carl review before merge per feedback_sensitive_code_review_before_merge.md.
