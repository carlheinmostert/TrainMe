Device QA — idempotent SQLite ADD COLUMN hotfix

Bug: TestFlight v1 install (1.0.0+1, build SHA f6f7bce) crashed with DatabaseException: duplicate column name: hero_crop_offset on first launch. App is unbootable past splash. Root cause is an upgrade-over-dev-install where the prior studio.homefit.app dev install left a Documents/raidme.db behind in a half-state — the column already existed but user_version was < 38, so the v38 migration tried ALTER TABLE exercises ADD COLUMN hero_crop_offset REAL and crashed.

Fix: local_storage_service.dart now routes every ALTER TABLE … ADD COLUMN through a new _addColumnIfMissing helper. Reads PRAGMA table_info, checks whether the column already exists, only ADDs if missing. Migration result is identical for healthy DBs (column added, version bumped); the half-state case now no-ops the redundant ADD and lets user_version stamp to 38 cleanly. All ~25 ADD-COLUMN sites in _migrateTables were retrofitted — not just v38 — so the next schema bump can’t put us back in this trap.

What did NOT change: _dbVersion stays at 38 (behaviour-preserving). _createTables is untouched (fresh installs don’t need this). RENAME COLUMN / DROP COLUMN in v33 untouched (different DDL, different semantics). No schema migrations applied cloud-side. No pubspec bump (Carl already bumped to 1.0.0+2).
