Device QA — Thumbnail re-publish after Hero drag

Bug. Moving the Hero star (or dragging the hero crop offset) on a video exercise's Demo / Preview tab regenerated _thumb.jpg / _thumb_color.jpg / _thumb_line.jpg locally, but the publish flow's fast-path keyed on rawArchiveUploadedAt alone and skipped ALL uploads — including thumbs. Cloud stayed on the stale autoPick=true motion-peak crop forever. Confirmed with curl: a portrait-source video showed 171×192px in cloud (the tight bbox) instead of the full-frame Hero offset the practitioner picked.

Fix. New local-only thumbnails_dirty column on the SQLite exercises table (DB v40). ConversionService.regenerateHeroThumbnails stamps it true after regenerating; UploadService.uploadPlan reads it three ways: (1) fast-path eligibility extended to rawArchiveUploadedAt && !thumbnailsDirty — any dirty exercise drops to the normal upload loop; (2) the normal loop's thumb-existence skip is overridden when dirty (uses upsert: true so re-upload is a silent overwrite); (3) the raw-archive _thumb_color.jpg upload also overrides its existence skip when dirty. Flag cleared post-success.

R-10 parity. Mobile-only change. The web player already reads the freshest cloud thumb on every load.

  • 3. Back in Studio, open the editor sheet for that video → Demo tab. Drag the Hero star along the timeline to a visibly different frame (e.g. mid-rep vs start). The Studio card's thumb updates immediately — that's the local regen. device
  • 4. THE fix. Re-publish. Pull-to-refresh (or fully reload) the Safari lobby URL. The card hero now reflects the new Hero offset you dragged to — not the previous frame. device

Hero crop offset drag → re-publish → lobby reflects

  • 6. Re-publish. Reload Safari lobby. The card crops to the new offset — the cloud thumb was re-uploaded with the new framing. device
