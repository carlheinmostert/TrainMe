Device QA — Members (Wave 5 + 14)

Purpose: verify /members identity surface + the Wave 14 add-by-email flow that superseded the Wave 5 invite-code flow. Owner types a colleague's email → if they have an account, added immediately; if not, a pending row is saved and auto-claims on signup via an auth.users INSERT trigger. Role-change + remove + leave flows unchanged.

In scope: /members table identity, add-by-email (three paths: added / already_member / pending), pending-row visibility + remove, auto-claim trigger on new signup, role changes with last-owner / self-role-change guards, remove, leave with last-owner / only-member guards, mobile surface cleanup.

Out of scope: audit log expansion (Wave 9), member avatars, in-app notifications of new joiners, Resend SMTP lift of the magic-link throttle (Wave 13, queued), email invites with a link. The auto-claim trigger fires on ANY new auth.users INSERT — Supabase handles whether the user got in via magic link / password / OAuth.

Prerequisites

  • Main merged through PR #97 (Wave 14). Supabase migration schema_milestone_u_add_member_by_email.sql applied live — pending_practice_members table + three RPCs (add_practice_member_by_email, remove_pending_practice_member, list_practice_members_and_pending) + claim_pending_practice_memberships_trigger on auth.users.
  • Two fresh Safari profiles so the owner session + test-user session don't collide (see "Safari → Settings → Profiles" if not already set up).
  • Access to at least one email address with no homefit.studio account yet (for the pending + auto-claim case). A second address that already has an account (for the instant-join case).
  • A practice where Carl is the owner with at least one other practitioner member, for role-change + remove + leave tests.

1 · Members list — identity surface

/members renders real identities (email + name) via list_practice_members_and_pending. Two sections on the page: Members (live) + Pending (added by email, not yet signed up).

  • 1. Open manage.homefit.studio/members as an owner — Members table shows a row per member with Email, Name, Role, Joined date. No truncated UUIDs.
  • 2. Own row is tagged "(you)" and has a Leave practice button visible in the Actions column.
  • 3. Open /members as a non-owner practitioner — Members table renders but Role dropdown, Remove button, Add-by-email input, and Pending section are all absent. Own row still shows Leave.

2 · Add by email — existing account (instant join)

Owner types the colleague's email, clicks Add. If the email matches an auth.users row, the practitioner is inserted into practice_members immediately.

  • 4. Type an email belonging to an existing homefit.studio user who is NOT yet a member of this practice. Click Add. Toast reads "Added {email} to {practice}." (or equivalent). The Members table gains a new row immediately.
  • 5. As that newly-added practitioner (in a separate Safari profile), refresh /dashboard → the practice appears in the switcher. No re-auth, no invite link clicked. Just appears.
  • 6. Try to Add the same email again. Toast reads "Already a member" (or equivalent — RPC kind=already_member). Members table does not duplicate the row.
  • 7. Try malformed input: empty string, single word with no @, blank spaces. Form should reject client-side with a sensible hint OR the RPC returns 22023 invalid email surfaced as a red toast. No 500 / blank page.

3 · Add by email — pending (no account yet)

When the email has no auth.users match, the RPC stashes a row in pending_practice_members. Pending section renders them.

  • 8. Type a fresh email with NO homefit.studio account (e.g. test-qa-{timestamp}@example.com). Click Add. Toast reads "Saved — will join automatically when they sign up" (or equivalent, kind=pending). The email appears in the Pending section, not the Members table.
  • 9. Pending row shows email + added_by (you) + added_at (now) + a Remove button. Fields match the mockup / component spec.
  • 10. Add the same pending email again. Idempotent — no duplicate row, added_at may refresh (ON CONFLICT DO UPDATE). Toast distinguishes this from the first-add case OR says the same "Saved" (either is fine; note which).
  • 11. Click Remove on a pending row → row disappears from the Pending section. Refresh the page — still gone. (RPC remove_pending_practice_member fired.)

4 · Auto-claim on signup (the trigger)

claim_pending_practice_memberships_trigger on auth.users INSERT drains matching pending rows into practice_members for the new user.

  • 12. Add a pending row for a fresh-never-seen email. In a separate Safari profile, go to manage.homefit.studio, sign up with that email (password or magic link). Confirm signup.
  • 13. Upon sign-in, the new user's practice switcher shows the practice that added them — no invite link, no extra click. Audit log (or practice_members direct query) shows a row with joined_at matching the signup time, role practitioner.
  • 14. Back in the owner's Safari profile, refresh /members — the email moved from Pending into the Members table.

5 · Role changes (owner ↔ practitioner)

Owner-only control, DB-enforced blocks on last-owner demotion and self-role-change. Unchanged by Wave 14.

  • 15. On a practice with 2+ owners, promote a practitioner to owner via the Role dropdown — update persists, table refreshes, their UI reflects owner permissions (Members page shows Add form + role/remove controls).
  • 16. Try to demote yourself — dropdown disabled / not present on your own row. If RPC is called directly via browser console, guard error (42501).
  • 17. On a practice where you are the only owner, try to demote another owner — blocked with "last owner — promote someone first" style message. No DB state change.

6 · Remove member + leave practice

Hard-delete the membership. Credit-ledger attribution preserved. Leave blocked for last-owner-with-practitioners and only-member cases.

  • 18. Click Remove on a practitioner row → row disappears. Their next portal refresh boots them out of that practice. credit_ledger entries attributed to them remain (FK to auth.users, not pivot).
  • 19. As a practitioner, click Leave on own row → removed. As the only owner with practitioners remaining, click Leave → blocked. As the only member, click Leave → blocked. All three paths work / block correctly.

7 · Mobile surface — invite-code cleanup

Wave 14 retired the mobile "Join a practice" card. Practitioners discover new practices via the switcher once the owner has added them.

  • 20. On iPhone, open Settings → scroll through the page top to bottom. The "Practices" card with the 7-char invite-code input is gone. No residual UI referencing "Join a practice" or "invite code".
