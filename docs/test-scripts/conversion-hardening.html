2026-05-14 · Conversion-pipeline hardening

Device QA — granular per-variant failure handling + eager backfill

Scope: three coordinated changes to the Flutter conversion service so thumbnail-variant failures become OBSERVABLE instead of silently swallowed. Stacked on the audit's no-silent-fallback principle (docs/audits/photo-video-treatment-audit-2026-05-13.md) — the parallel agent ships placeholder rendering on the resolver side; this PR ensures missing variants get logged + eagerly regenerated.

Change 1 · Granular per-call try/catch. Post-conversion thumbnail block + regenerateHeroThumbnails previously wrapped the three extractFrame calls (B&W, color, line) in ONE outer try/catch. A failure on extract #2 swallowed extract #3 silently and left only _thumb.jpg on disk. Now each variant has its own try/catch; one failure doesn't poison the others.

Change 2 · _logVariantFailure + _logBackfillEvent. New per-variant log helpers append to {Documents}/conversion_error.log with a distinct [VARIANT bw/post_conversion] / [BACKFILL start line] prefix on the timestamp line. The existing failed-pill long-press sheet (PR #213) parses the new tag and renders a coral / sage chip per entry so a thumbnail-variant miss is visually distinct from a full conversion failure.

Change 3 · Eager backfill on session open. ConversionService.backfillMissingVariants(exercises) walks every done exercise on the session and re-runs extractFrame for any of _thumb.jpg / _thumb_color.jpg / _thumb_line.jpg that's missing on disk. Fires from StudioModeScreen.initState, fire-and-forget. Per-variant granular try/catch; each successful regen stamps the exercise and emits on ConversionService.onConversionUpdate so existing listeners refresh.
