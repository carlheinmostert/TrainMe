Wave 36 · device QA

Avatar EXIF orientation + 21×21 count badge + photo body-focus end-to-end

Three carry-over fixes from Wave 34 device QA. (A) Captured avatar PNG arrived 90° rotated even though the live preview was upright. The W34 fix pinned videoOrientation at session setup, but iOS can re-rotate connections during interface-orientation transitions. Wave 36 force-pins portrait at every capturePhoto call (with os_log diagnostic for the pre/post values) AND bakes EXIF orientation into the pixel buffer inside ClientAvatarProcessor so Vision + vImage operate on an already-upright source. The output PNG no longer relies on EXIF metadata that PNG can't carry. (B) Wave 34 shrunk the count badge 16×16 → 14×14 to fit "naturally" against the +50% leading icon — QA found the result anemic. Wave 36 reverses direction: 14×14 → 21×21 (+50%), font 9/7 → 13/10. Anchor offset moves -3/-3 → -5/-5 so the larger badge still feels pinned to the corner. (C) Photo Body Focus pill went all the way to chrome but never had a segmented JPG to swap to. Wave 36 ships the full pipeline: native processPhotoBodyFocus method on VideoConverterChannel (reuses ClientAvatarProcessor as JPEG), conversion service writes {exerciseId}.segmented.jpg alongside the line drawing, publish uploads to raw-archive, get_plan_full emits grayscale_segmented_url + original_segmented_url for photos, web player flips between raw & segmented when body-focus is on.

A · Avatar still upright orientation

  • 1. Capture an avatar in portrait — output is upright, head at top.
    1. Open any client without an avatar. Tap the avatar circle → "Take photo".
    2. Frame the subject normally with the phone in PORTRAIT (default). Tap the shutter.
    3. After the ∼1s spinner, the body-focus PNG appears as the avatar circle. Subject's head is at TOP, feet at BOTTOM. NOT rotated 90° left like W34 #1 fail. (Background should be Gaussian-blurred, body crisp — same look as before; only orientation changes here.)
    4. Re-open the client → the avatar is still upright (no transient rotation that "fixes itself" later).
  • 2. Hold the phone landscape briefly during framing — output is STILL upright (portrait-locked).
    1. Open the avatar capture screen. The screen is portrait-locked via OrientationLockGuard; the live preview should NOT rotate when you tilt the phone.
    2. Hold the phone in landscape for ∼2s while pointed at a subject, then bring it back to portrait, then tap shutter.
    3. The captured PNG is still upright (head at TOP). The connection re-pin at capturePhoto-time should defeat any landscape-kick the OS tried to apply mid-session.
    4. If you have access to Console.app, filter on subsystem com.raidme.raidme + category avatar.capture — you should see a "connection videoOrientation pre=N post=1" log line at every capture. (1 = portrait.)

B · Count badge 21×21 (50% larger)

  • 3. Session-card count badge reads as a confident accent next to the +50% icon — not anemic, not crowding.
    1. Open a client with at least one published session. Look at the session card row.
    2. Bottom-right of the leading list-alt icon: a coral pill with the exercise count. Single-digit (e.g. "3") sits centered with breathing room. 2-digit ("12") + the "99+" stadium variant both fit cleanly inside the new 21×21 footprint.
    3. Compare against a session with 0 exercises: badge HIDDEN entirely (no empty pill).
    4. Visual sanity — the badge feels pinned to the icon's bottom-right corner, not floating off it. Anchor offset is -5/-5 at this size.

C · Photo body-focus end-to-end

  • 4. Capture a NEW photo → segmented JPG is generated locally during conversion.
    1. Open Studio on any session. Tap a photo capture (NOT a video) for a person doing an exercise.
    2. Wait for the conversion-status pill to clear ("∼1 converting..." disappears). On Pro-class iPhones this should be < 1.5s additional vs the line-drawing alone.
    3. If you have Xcode + Console.app access: there should be NO "Photo body-focus segmentation failed" log lines for this exercise. (A failure is non-fatal; the line drawing still ships, but body-focus would be empty for that exercise.)
    4. Conversion succeeds — the line-drawing thumbnail renders normally on the Studio card.
  • 5. In-app Preview — Body Focus pill swaps source visibly on PHOTO slides.
    1. In the same Studio, tap the photo capture → opens the _MediaViewer. Per-exercise treatment defaults to Line.
    2. Swipe to Original (or B&W). The image swaps to the raw colour photo. Body Focus pill becomes ENABLED (not greyed).
    3. Tap Body Focus pill — the photo swaps to the segmented variant: SUBJECT's body crisp, BACKGROUND visibly Gaussian-blurred. Same look as the avatar surface.
    4. Tap Body Focus again — reverts to the untouched raw colour photo. (Toggle is sticky per device via SharedPreferences; stays on between sessions.)
    5. Switch to B&W — segmented variant + grayscale CSS filter both apply (body crisp & B&W, background blurred & B&W).
  • 6. Web player — Body Focus toggle on a photo slide swaps source over the wire.
    1. Confirm the linked client has consented to grayscale + original (Studio → client detail → consent toggles, or via the publish-time consent prompt).
    2. Publish the session. Tap the share icon → copy the URL.
    3. Open the URL on a different device or in Safari Private. Navigate to the photo slide.
    4. Use the "Show me" segmented control to switch to Colour. The body-focus toggle in the gear popover is ON by default.
    5. Toggle Body Focus OFF: image swaps to the untouched raw colour JPG (background un-blurred). Toggle back ON: returns to the segmented JPG. Web player version footer reads v68-photo-body-focus.
  • 7. Line treatment is unaffected — Body Focus toggle does NOTHING on Line (already abstract).
    1. On the photo slide (in-app Preview AND web player), switch to Line treatment.
    2. Body Focus pill greys out (disabled). Tooltip / disabled state explains "Body focus applies to colour playback only."
    3. Image renders the line-drawing JPG as before — black-and-white outline aesthetic, no Gaussian blur.

Out of scope — reminder

  • Backfill of segmented JPGs for legacy photos — only NEW captures (post-Wave-36) generate the segmented file. Old photos still play their raw + line-drawing only; the body-focus toggle is a no-op for them. Re-capturing the photo will produce a segmented variant on the next conversion.
  • Avatar EXIF fix is upstream of Vision — old wonky avatars stored before W36 are still rotated. Re-capture to get an upright PNG.
