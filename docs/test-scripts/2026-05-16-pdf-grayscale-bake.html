Device QA — PDF honours B&W treatment per row

Bug. In the published-plan PDF, every row rendered as colour even when the practitioner had set rows to the B&W treatment. The live lobby honoured the per-row treatment correctly. Carl spotted this while QAing the post-PR #364 crop fix — the crop itself was correct, but flipping a row to colour appeared to flip the entire PDF preview.

Root cause. The B&W treatment is realised by a CSS rule (img.is-grayscale { filter: grayscale(1) contrast(1.05); }) applied at display time on top of the colour source JPG. html2canvas (the PDF rasteriser) ignores CSS filters on <img> tags during snapshot — same class of bug PR #364 fixed for object-fit: cover. Result: every B&W row rasterised as the underlying colour bitmap.

Fix shape. Bake the grayscale into the bitmap inside the hero resolver's canvas op (rasteriseToSquare). When the treatment is bw, set ctx.filter = 'grayscale(1) contrast(1.05)' before the drawImage. The cache key already includes treatment, so a bw row gets a grayscale data URL and an original row gets a colour data URL with no extra round-trip. The CSS class stays harmlessly on the live lobby (no-op on an already-grey bitmap).

R-10 parity. web-player/hero_resolver.js and app/assets/web-player/hero_resolver.js byte-identical post-edit. Both surfaces (live web player + embedded WKWebView in mobile Studio Preview) get the fix.

  • Build from fix/pdf-grayscale-bake-into-resolver deployed to staging.session.homefit.studio AND a fresh install of the staging Flutter app on Carl's iPhone via ./install-device.sh staging (so the embedded WebView gets the mirrored asset).
