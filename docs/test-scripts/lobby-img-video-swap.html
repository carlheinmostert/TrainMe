Device QA · iPhone · Lobby img/video swap (v48)

Lobby img/video swap — structural single-video guarantee

v47 (PR #251) tried to keep one <video> per row and manage src on/off + load() based on active state. Carl confirmed on device that multiple videos still played simultaneously — iOS WKWebView either doesn't honour removeAttribute('src') quickly enough, or some path re-engages playback on inactive rows.

v48 sidesteps the issue entirely: inactive rows are <img> elements (no decoder allocation possible). When a row becomes active, its <img> is replaced with a freshly-created <video> via parentNode.replaceChild. When inactive, swapped back to <img>. The orphaned <video> is GC'd and its decoder slot fully reclaimed because the element no longer exists in the document.

Architectural commitment: at any moment, the lobby has 0 or 1 <video> elements in the DOM. Never more.

SW cache bumped: v47-lobby-single-active-video → v48-lobby-img-video-swap. Build chip should read v48-lobby-img-video-swap on both bundle & cache.

Circuit-breaker instrumentation from PR #250 is unchanged. The [CB-hb] heartbeat is still your proof that the JS event loop is alive.

  • Pull / hard-refresh / kill+relaunch the dev app so the new bundle activates. Confirm the build-marker chip reads v48-lobby-img-video-swap.
  • Connect iPhone to Mac via USB. Open Safari (desktop) → Develop → [Your iPhone] → pick the WebView entry under the dev app process.
