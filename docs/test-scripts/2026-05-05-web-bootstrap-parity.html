Device QA — web-portal practice bootstrap parity

Scope: close the auth-flow parity gap between the mobile Flutter app and the manage.homefit.studio web portal. The mobile AuthService calls the idempotent SECURITY DEFINER RPC bootstrap_practice_for_user() on every authenticated sign-in — the web portal did not, so first-time web sign-ups landed on the dashboard’s “not yet a member of any practice” empty state.

Fix: added PortalApi.bootstrapPractice() wrapper, called from THREE entry points:
1. /auth/callback route handler (covers magic-link sign-in/sign-up after Supabase round-trip).
2. SignInGate after successful signInWithPassword (password path bypasses the callback).
3. SignUpGate after successful signInWithPassword (same reason).

Best-effort everywhere: failures log + fall through to dashboard, RPC is idempotent so the next sign-in retries.

1 · Magic-link first sign-up creates practice

  • 1. On the deployed portal, sign up via magic link. Open /sign-up (or root with no session), enter fresh email A, leave password blank, hit Continue. Tap the magic-link in the email. Browser redirects through /auth/callback?code=... and lands on /dashboard. The dashboard renders the 4–5 stat tiles (NOT the “not yet a member of any practice” empty state).
  • 2. In the Supabase SQL editor, run the verification query (substitute email A) and confirm a practice + membership + welcome-bonus credit row exist. 
    SELECT
  u.id          AS user_id,
  u.email,
  pm.practice_id,
  p.name        AS practice_name,
  p.owner_trainer_id = u.id AS is_owner,
  cl.delta      AS welcome_credits,
  cl.type       AS credit_type
FROM auth.users u
LEFT JOIN practice_members pm ON pm.trainer_id = u.id
LEFT JOIN practices p          ON p.id = pm.practice_id
LEFT JOIN credit_ledger cl     ON cl.practice_id = p.id AND cl.type = 'adjustment'
WHERE u.email = 'EMAIL_A_HERE';
-- Expect: 1 row. is_owner=true, welcome_credits=5, credit_type='adjustment'.
-- practice_name should be "{local-part} Practice".
