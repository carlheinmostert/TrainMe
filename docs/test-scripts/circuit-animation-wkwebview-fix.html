Device QA - Circuit animation in embedded WKWebView (E14)

Background. The lobby's coral "tracer" spiral spins around the circuit frame in Mobile Safari at staging.session.homefit.studio - but renders static in the iOS embedded workflow Preview (same web-player bundle, served via the homefit-local:// custom scheme handler inside a WKWebView).

This is the third attempt at E14.
- PR #317 (attempt #2) diagnosed root cause as getTotalLength() returning 0 before WKWebView's layout pass; added a single requestAnimationFrame retry + try/catch around WAAPI. Carl QA'd - still static.
- This attempt (#3) reasons further: the CSS animation declaration on .lane-tracer referenced an undefined var(--path-len) in its keyframes, producing an invalid keyframe. On iOS WebKit with the element inside a position: fixed ancestor (the lobby IS position: fixed), the CSS animation's no-op keyframes can suppress the layered WAAPI animation's visible effect.

Fix - four layers of defence:
- (a) Disable the CSS animation explicitly: tracer.style.animation = 'none'. Removes the CSS-vs-WAAPI cascade-priority fight entirely.
- (b) Also remove the broken CSS keyframes / @media (prefers-reduced-motion) rule from styles.css. WAAPI is the sole source of truth now. Reduce-motion is honoured in JS.
- (c) Replace the single rAF retry with a bounded poll loop (up to 30 frames / ~500ms) that re-reads getTotalLength() each frame until it returns a positive number, then starts the animation.
- (d) After WAAPI animate() returns, verify anim.playState isn't idle; if it is, explicitly .play() + re-check on the next frame; settle gracefully if still inert.

R-10 parity: all web-player/* edits synced to app/assets/web-player/ via the bundle sync tool. Byte-identical.

In scope: embedded preview circuit lane animation - tracer drawing motion. Live web player regression check.

Out of scope: lobby export (Task D in round 2), prep hero treatment, photo body-focus, filmstrip sharpness - all separate PRs.

Reduce Motion accessibility setting

The embedded preview is practitioner-facing (not client-facing). The brief asked to confirm one specific design call: should the embedded surface respect the device-wide Reduce Motion preference, or always animate? This fix honours Reduce Motion uniformly across both surfaces - the same JS branch checks window.matchMedia('(prefers-reduced-motion: reduce)').matches in applyTracer. Rationale: consistency-first; a practitioner who needs Reduce Motion still benefits from honouring it in preview.

  • 6. Open Settings → Accessibility → Motion → turn Reduce Motion ON. Re-open the embedded preview lobby. The circuit tracer renders as a static settled spiral (stroke-dashoffset 0 - the full path drawn, not animating). Lane outlines still visible.
  • 7. Turn Reduce Motion OFF. Re-open the embedded preview lobby. Tracer animates again. (No fresh build required; the matchMedia query re-evaluates on each lane render.)
