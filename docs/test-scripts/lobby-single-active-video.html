Lobby single-active-video — dodge HW decoder starvation

PR #250's circuit-breaker confirmed the lobby freeze is host-side, not a JS runaway: heartbeat dies with no [CB] trip, JS counts stay low. HW H.264 decoder starvation in embedded WKWebView fits the signature — the lobby played current ± 1 (up to 3 videos simultaneously) and iOS allocates only ~3–4 concurrent decoders to embedded WebViews.

Fix in v47: the active row plays its hero video; every other row shows the cropped Hero frame as a static poster= image. Inactive rows clear src + call load() to fully release the HW decoder (pause alone keeps the slot reserved). 150ms debounce on the kick during fast scroll so the active idx churn doesn't thrash the decoder.

Same lesson as Studio cards (PR #218): multiple video heroes on one surface = thermal / decoder pressure. Active-only playback is the established pattern; the lobby was the outlier.

SW cache bumped: v46-lobby-circuit-breaker → v47-lobby-single-active-video. Build chip should read v47-lobby-single-active-video on both bundle & cache once the new SW takes over.

The diagnostic still works: the circuit-breaker from PR #250 is still wired; the heartbeat is the proof point. If it stays alive through sustained scroll on v47, decoder starvation is confirmed and fixed. If it still dies, decoders were a red herring and we pivot.

Prerequisites

  • Pull / hard-refresh session.homefit.studio in mobile Safari (kill app + relaunch) so the new SW activates. Confirm the build-marker chip reads v47-lobby-single-active-video on both bundle & cache (divergence = stale SW).
  • Connect iPhone to Mac via USB. Open Safari (desktop) → Develop → [Your iPhone] → session.homefit.studio. Switch to the Console tab. The heartbeat lines from PR #250 are how you'll know the JS event loop is still alive.
  • For the WebView path (Workflow Preview): open the Flutter app on your iPhone, drive a session into Workflow Preview, then in desktop Safari Develop → [iPhone] choose the WebView entry under the Flutter app process.
  • Plan should have at least 8–10 video rows so there's enough scroll runway and enough heroes to test the swap.

1 · Build is what we think it is

  • 1. Cache chip reads v47. The version chip in the lobby footer should say v47-lobby-single-active-video. If it still reads v46- or v45-, the SW is stale — force-reload (kill app, clear, relaunch) until it advances. Bundle + cache must match.
  • 2. Heartbeat appears in console. Within ~4 seconds of the lobby rendering, [CB-hb] {…} lines appear every ~2s in the Safari Web Inspector console (carried over from PR #250). This is the proof that the JS event loop is alive. If you see no heartbeat at all, the SW didn't update or showLobby didn't run.

2 · Single-active-video looks right

  • 3. Only ONE video plays at a time — the active (centered) row. Sit on the lobby and look at all the rows: only the row tracked by the matrix coral pill should have visible motion. Every other row should display its cropped Hero frame as a crisp static image (the poster= behind the <video>). NO black frames, NO thumbnail jank, NO multiple rows playing simultaneously.
  • 4. Motion follows your focus as you scroll. Slowly scroll the lobby. As you settle on a new row, the previous row's video freezes back to its Hero poster (within ~150ms after the scroll settles — that's the debounce), and the new active row's video starts playing. Expected: the "live" feeling moves with you; everything else is a still life. Failure mode: two videos visibly playing at once, or the previous row stays animated after you've moved past it, or the new active row never kicks.
  • 5. Fast fling-scroll doesn't visually churn. Fling-scroll the lobby quickly past several rows without settling. Expected: while in motion, NO video kicks in (the 150ms debounce eats every transient active-row change); all rows show their static Hero poster. Once you settle on a row, that one row's video starts after a beat. Failure mode: rapid play/stop flicker on intermediate rows, OR the wrong row plays after the scroll settles.

3 · The freeze is gone

  • 6. Sustained scroll for 30+ seconds — NO freeze. Scroll up and down continuously for at least 30 seconds with finger sustained on the screen for several seconds at a time (the exact pattern that reliably reproduced the freeze on v46). Expected: heartbeat stays alive throughout ([CB-hb] lines every 2s, never going silent). NO black-out. NO [CB] runaway warnings. The lobby remains responsive throughout. This is the load-bearing item — if it passes, decoder starvation is confirmed as the root cause and fixed.
  • 7. If the freeze STILL happens (it shouldn't): capture the LAST 5 lines of console output before the heartbeat dies. Note exactly which counters were active and what their counts were. We'd then pivot to memory-pressure or scheme-handler hypotheses. Mark this PASS if no freeze, FAIL with detail if it does happen.
