Device QA — Line-drawing prefetch on Studio open

Purpose: verify that opening a cloud-only session in Studio downloads the line-drawing video in the background, the per-card spinner overlay surfaces progress, and the missing-media banner clears once the file lands. Stacks on top of PR #190 — sessions hydrate from cloud but their media is still on Supabase storage.

In scope: new MediaPrefetchService singleton; list_practice_plans RPC extended with explicit line_drawing_url; per-exercise ValueNotifier<MediaPrefetchStatus>; coral "Downloading line drawing…" chip replacing the red "Media missing" chip while a pull is in flight; SQLite converted_file_path updated atomically (.partial → rename) on completion; Studio refreshes session in-memory so MiniPreview picks up the new file without a manual reload.

Out of scope: B&W and Original treatments (still go through the existing on-treatment-switch download path against raw-archive with signed URLs); background prefetch on Home; "Download all" affordance; explicit retry button.

Prerequisites

  • Branch claude/clever-euler-1d2a03 installed via the worktree-build incantation in the brief (NOT install-device.sh — that pulls main).
  • Carl's iPhone 00008150-001A31D40E88401C, unlocked.
  • At least one practice with cloud-published sessions whose media is in the public media bucket.
  • Stream device logs in a terminal: idevicesyslog --udid 00008150-001A31D40E88401C 2>&1 | grep -E 'MediaPrefetchService|SyncService|prefetchSession'.

1 · Cold-launch hydration still works (regression check on #190)

The prefetch is additive — the cloud→local sessions branch from PR #190 should still pull plans and render the clients list with non-zero session counts.

  • 1. Cold-launch the freshly-installed app. Sign in. os_log shows SyncService: pullSessions practice=<uuid> — cloud=N inserted=N skipped_existing=0 and the clients list renders with non-zero session counts under each client. Tap into a client → ClientSessionsScreen renders the full pulled list.

2 · Open a cloud-only session — auto-download replaces the red banner

The headline test. Tap a pulled session → Studio mode → cards transition from Media missing → Downloading line drawing… → playable, with no user intervention.

  • 2. Tap a pulled session card. Studio opens; for every video / photo exercise the card initially shows the red "Media missing — long-press to recapture" chip. Within ~1 second of Studio opening, os_log shows MediaPrefetchService: prefetchSession session=<uuid> — queued M downloads (cap=3). The red chips flip to coral "Downloading line drawing…" chips with a small spinner.
  • 3. Within ~5-30 seconds (depends on connection + plan size), each card's coral chip vanishes as its file lands. The MiniPreview starts playing the line-drawing video flood-filled in the card. No red chip remains for any video / photo card. Rest cards are unaffected (no chip, no spinner — the prefetch skipped them).
  • 4. Open Preview on the same session. The plan-preview deck renders. Each card plays the freshly-downloaded line-drawing treatment — same content as the Studio MiniPreview shows, no fall-through to a placeholder. Rep-stack and progress pills track normally.

3 · Network drop mid-download leaves the red banner, no crash

Failures must NOT loop or cascade. The card falls back to "Media missing" so the practitioner can long-press to recapture.

  • 5. Find a fresh cloud-only session that hasn't been opened yet (from a different client, or after force-uninstall + reinstall). Turn on airplane mode. Tap the session. Studio opens; coral chips appear briefly. Within ~5-15 seconds the chips on every card flip back to red "Media missing — long-press to recapture". os_log shows one MediaPrefetchService: _runTask failed exerciseId=<id>: … per exercise. App does NOT crash, hang, or throw a SnackBar.
  • 6. With airplane mode still on, navigate back out of Studio and re-open the same session. No second flurry of failed downloads — os_log shows the prefetch ran once per session per app launch (the in-memory _scheduledSessions guard). To re-try, the user must restart the app (with network back) — the session re-opens fresh and pulls succeed.

4 · Re-opening a successful session is a no-op (idempotent)

  • 7. From the session that completed in item 3 (every file landed locally), navigate back to the clients list and re-open the session. os_log shows MediaPrefetchService: prefetchSession session=<uuid> — nothing to download within the same app run. Cards play immediately; no spinners, no chips. Force-quit the app and re-launch + re-open the same session — os_log again shows "nothing to download" because the local files exist on disk.

Patch on top of the prefetch wave: tapping a locked B&W or Original segment now disambiguates between "raw not on disk" (opens the download sheet) and "client hasn't consented" (SnackBar names the client + treatment).
