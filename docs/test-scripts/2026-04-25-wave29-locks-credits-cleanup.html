A · Structural-edit lock + unlock affordance

  • 1. Padlock icon hidden when plan is unopened OR within 3-day grace.
    1. Publish a fresh plan. Don't open the URL on Mac Safari (so first_opened_at stays NULL).
    2. Reopen the session in Studio. Top action bar: padlock icon NOT visible. Add/delete/reorder all enabled.
    3. Open the URL on Mac Safari (stamps first_opened_at). Force-close + reopen Studio. Within 3 days the padlock should still be HIDDEN.
  • 2. Padlock icon visible after 3 days post-first-open.
    1. Use a plan whose first_opened_at is > 3 days ago (or use supabase db query --linked to backdate one).
    2. Open the session in Studio. Padlock icon visible right side of top action bar (coral on dark pill).
    3. Add/delete/reorder affordances are visually disabled (the existing locked state on _InlineActionTray).
  • 3. Tap padlock → confirm sheet → Unlock consumes credit + unlocks Studio.
    1. Tap the padlock on a locked plan. Sheet appears titled "Unlock plan for editing" with the cost-1-credit copy + version stays at vN.
    2. Tap "Unlock" (coral). Credit balance on the home chip drops by 1. Sheet closes. Studio re-renders with edit affordances enabled, padlock GONE.
    3. Make a structural change (delete an exercise). Re-publish. Verify NO additional credit charged (the prepaid one covers this republish). Subsequent edits + republishes after THIS go back to normal lock-window rules.
  • 4. Cancel on the unlock sheet leaves state unchanged.
    1. Tap padlock → sheet opens. Tap "Cancel" (or backdrop swipe-down).
    2. No credit consumed. Studio still locked. Padlock still visible.
  • 5. Insufficient credits at unlock → standard insufficient-credits sheet.
    1. Drain a practice's credits to 0 (or use a fresh practice with no purchases yet).
    2. Tap padlock on a locked plan → tap "Unlock". The standard "not enough credits" sheet appears with a "Top up" link to portal.
    3. No state change on the plan; no ledger row written.
  • 6. Schema spot-check: plans.unlock_credit_prepaid_at + clients.consent_confirmed_at exist and round-trip.
    1. Optional: supabase db query --linked "select id, unlock_credit_prepaid_at from plans where unlock_credit_prepaid_at is not null limit 5;"
    2. After unlocking + republishing in #3, the column should be NULL again (the publish path cleared it).
