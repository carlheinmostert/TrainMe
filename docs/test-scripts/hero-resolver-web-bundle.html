2026-05-13 · Hero Resolver — Bundle 1 of audit

Device QA — Exercise Hero Resolver (Bundle 1: B6 + D13)

Purpose: Bundle 1 of the photo-vs-video treatment audit migration. Introduces web-player/exercise_hero.js — a single stateless resolver every web surface (lobby strip · deck active slide · prep overlay · share-as-PNG snapshot) calls to pick treatment-correct src + poster + body-focus caps. Targets two open QA failures from the audit:

Audit fixes in this PR:
· B6 part 1 (lobby leak) — Setting the first video to Line caused EVERY lobby row to render in Line (B&W rows leaked too). Was: single lobby-global activeTreatment driving every row's renderHeroHTML call. Now: per-row resolver call with each exercise's own effective treatment.
· B6 part 2 (prep hero treatment) — Prep-phase Hero was always the legacy thumbnail_url (B&W from raw for videos), so Line treatment showed B&W during the countdown then jumped to Line on play. Now: buildPrepOverlay calls the resolver with surface: 'prep' and reads posterSrc + domClass.
· D13 (snapshot PNG poster) — <video poster> does NOT inherit CSS filters during html2canvas rasterisation, so the active video row's poster in the B&W snapshot PNG showed up as colour (or wrong-treatment). Now: HomefitHero.bakeFilterIntoDataUrl bakes the filter into the poster's data URL BEFORE html2canvas reads it.
· Active deck slide — buildMedia's posterAttr was always exercise.thumbnail_url regardless of treatment, so Line/Original playback flashed a B&W poster during decoder warm-up. Now: resolver-picked.

Stacked for later bundles (do NOT regress on these but they don't need to be FIXED here):
· F17 (Flutter filmstrip — 3 photos render as 1 cell)
· F21 (photo thumbnail variant pipeline + cloud upload)
· PR 4 (Flutter resolver — Studio cards / MediaViewer / filmstrip migration)
· Cloud-surface D13 polish — public web player only has thumbnail_url on the wire; thumbnail_url_line + thumbnail_url_color arrive with Bundle 2's PR 6 (photo variant pipeline). Until then, Line/Original snapshots on cloud-fetched plans may still fall back to the legacy thumbnail. Embedded preview gets the full fix immediately since the bridge already emits all three URL fields.

R-10 parity: web-player/exercise_hero.js + all touched files synced into app/assets/web-player/ via dart run tool/sync_web_player_bundle.dart.

In scope: lobby strip per-row treatment · prep overlay hero treatment · active deck slide poster treatment · share-as-PNG poster bake · embedded preview parity · audit doc reference.

Out of scope: Flutter filmstrip · mini_preview · studio_mode_screen MediaViewer · conversion_service photo variant pipeline · upload_service photo thumbnails · web-portal · referral / payments paths.

Prerequisites

  • Open the staging branch by navigating to staging.session.homefit.studio/p/<planId> on the iPhone Safari. Confirm the build chip in the bottom-right shows the new branch SHA before testing.
  • For B6: published staging plan with at least THREE exercises in three different preferred treatments — one Line, one B&W, one Original (consent permitting). Easiest setup: open Studio, set per-exercise treatment via the segmented control on each card, then publish.
  • For D13 (snapshot): same plan, plus at least ONE video exercise with B&W preferred treatment. The snapshot bake only kicks in for B&W on videos (Line + Original treatment URLs are already correct on disk; B&W needs the CSS filter baked in).
  • For embedded preview: Studio → Preview button on the same plan. iOS WKWebView loads the mirrored bundle from app/assets/web-player/.

Task 1 · Lobby per-row treatment (audit B6 part 1)

Different exercises with different per-exercise preferred_treatment show different treatments in their respective lobby rows. Setting the first exercise to Line no longer leaks Line into every other row.

  • 1. Open the test plan in Safari at staging.session.homefit.studio/p/<planId>. The lobby renders the row strip. The Line exercise's row shows the line-drawing Hero (white outline on dark bg). The B&W exercise's row shows the grayscale Hero. The Original exercise's row shows the colour Hero. Three different visual treatments visible side-by-side in the lobby — no single treatment dominates.
  • 2. Set the FIRST non-rest exercise to Line treatment in Studio, then re-publish. Open the published plan in Safari again. The first row renders Line, but the OTHER rows still render in their own treatments — NOT leaked to Line. (Pre-fix: every row would render Line because of the lobby-global activeTreatment seeded from the first exercise.)
  • 3. Tap the lobby gear (top-right of CTA bar) → tap a treatment pill (eg. B&W). The lobby's plan-global override fires — every row re-renders in B&W. (This is the applyTreatmentOverrideToAllExercises path; it writes a per-exercise override on every slide BEFORE the resolver runs, so the resolver sees B&W as the effective treatment everywhere.) Tap Line — every row flips to Line.

Task 2 · Prep overlay hero respects treatment (audit B6 part 2)

The Hero shown behind the 15-second prep countdown matches the upcoming exercise's effective treatment. No more colour-flash before B&W playback or B&W-flash before Line playback.

  • 4. Tap Start workout. Prep countdown begins (15 → 1). The Hero behind the digit matches the upcoming exercise's effective treatment:
    · If the first exercise is Line → Hero is line-drawing (white outline). No grayscale.
    · If B&W → Hero is grayscaled (CSS filter: grayscale(1) contrast(1.05)).
    · If Original → Hero is full colour.
    When the countdown finishes, the video continues in the SAME treatment with no jarring colour-to-X swap.
  • 5. Navigate (swipe / chevron) to the NEXT exercise. The 15s prep restarts (or shortens if mid-workout). The new Hero behind the countdown matches THIS exercise's treatment, not the previous one's. Repeat for all three exercises (Line, B&W, Original) — each one's prep Hero matches its own treatment.
  • 6. Open the gear popover mid-workout → flip the treatment for the current exercise (or globally via the lobby pill before re-entering). Re-trigger the prep (next exercise / next slide). The prep Hero now reflects the NEW treatment, not the old one. Treatment changes are picked up on the next render — no stale poster.

Task 3 · Active video poster respects treatment

The <video poster> attribute on the active deck slide (visible during decoder warm-up, between treatment swaps, etc.) matches the active treatment.

  • 7. Open a Line-treatment exercise on the deck. Briefly pause via the timer chip then resume — the freeze-frame between unloads shows the line-drawing Hero (NOT B&W). Repeat on a B&W exercise — freeze-frame is B&W. On Original — freeze-frame is colour.
  • 8. Flip treatment via the gear popover (eg. B&W → Original on a video). During the brief swap window before the new video URL loads, the poster shown is the NEW treatment (Original — colour). Not the old one. (On the public surface where only thumbnail_url is on the wire, Line / Original still fall back to the legacy B&W thumb until Bundle 2's PR 6 — that's expected, document it as a "pass with note" if seen.)

Task 4 · Share-as-PNG snapshot bakes B&W into video poster (audit D13)

The lobby share button generates a PNG snapshot. Without the bake, the active video row's poster in the PNG would render in colour (or wrong treatment) because <video poster> doesn't inherit CSS filters during html2canvas rasterisation. Now the filter is baked into the poster URL's pixel data BEFORE html2canvas reads it.

  • 9. In the lobby, set treatment to B&W (gear → B&W pill, every row flips). Tap the share button (top-right of CTA bar). Modal appears with spinner; within ~3 seconds, the rendered PNG replaces the spinner. In the PNG, the ACTIVE video row's poster (the row currently in view) is rendered in B&W — grayscale tones, no colour bleed. Inactive rows are also B&W (those use <img> + .is-grayscale which html2canvas onclone handles correctly).
  • 10. Flip back to Line treatment in the lobby. Tap share again. The PNG renders the active video row's poster as the line-drawing Hero (white outline, no B&W filter, no colour). Inactive rows match. (On the public web surface only — embedded preview gets the line variant from the bridge; cloud may fall back to legacy until Bundle 2's PR 6.)
  • 11. Flip to Original. Tap share. The PNG renders the active video row's poster in colour. Photos in the lobby (if present) also render in their respective treatments — Line photos = line drawing, B&W = grayscaled raw, Original = full colour.
  • 12. Trigger a failure path: if toBlob returns null OR the canvas is tainted (shouldn't happen on staging, but it's the safety net), the modal swaps the spinner for a red error message — NOT a silent hang. Confirm the error path is still reachable by inspecting preloadErrors in the console after a share (should be empty in the happy path).
