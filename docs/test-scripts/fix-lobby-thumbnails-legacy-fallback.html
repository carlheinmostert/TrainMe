2026-05-15 · Lobby hero thumbnails — legacy plan soft-fallback

Device QA — Empty Hero Thumbnails on Legacy Plans

What broke: after PR #348 swapped the default treatment from line to B&W, plans created before PR #319 (2026-05-13 photo variant pipeline) opened in the embedded preview with empty grey hero thumbnail squares. Those legacy plans only have a single thumbnail_url file on disk — neither _thumb_color.jpg nor _thumb_line.jpg was generated when they were captured. The iOS scheme bridge emits all three URLs unconditionally, so WKWebView resolves the missing variant to fileDoesNotExist and the <img> renders blank.

What landed: JS-side soft fallback in pickPosterSrc (web-player/exercise_hero.js). When the requested variant URL isn't present, we fall back to the canonical thumbnail_url. The CSS .is-grayscale filter still applies to the <img>, so B&W still looks B&W — it's just generated from the canonical thumbnail rather than the missing variant.

In scope: embedded preview lobby on legacy plans (created before today, 2026-05-15). Same preview on fresh plans (created today) for regression check.

Out of scope: public web player surface (uses get_plan_full RPC which only returns one thumbnail_url; this code path isn't exercised); video playback (this fix is thumbnail-only).
