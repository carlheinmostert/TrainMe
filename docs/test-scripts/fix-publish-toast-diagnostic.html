2026-05-15 · publish-toast-diagnostic

QA — publish toast diagnostic + recovery copy polish

Scope: app/lib/services/upload_service.dart (5 loudSwallow upload sites) + app/lib/screens/studio_mode_screen.dart (one toast string).

Why this exists: the “Published, but some optional treatment files are still processing (raw-archive upload incomplete). Line treatment is live now; retry publish later to backfill.” toast fires on every publish on staging, but doesn't name which file is failing. The generic loudSwallow rejection swallows the error string, leaving the practitioner with no signal beyond “something didn't upload.”

Diagnostic patch: each variant loudSwallow block (main mp4, segmented mp4, photo color, color thumb, mask) now appends local_path + file_exists bool to its meta map, AND emits a debugPrint line on failure with kind + path + exists. Surfaces in Xcode device console or via the failed-pill long-press conversion error log.

Copy polish: the toast now reads “Published. Some optional treatment files are still processing — retry publish in a minute to backfill the rest. Line treatment is live now.” (drops the parenthetical “raw-archive upload incomplete” jargon; adds a concrete recovery path).

Install: after the PR merges to staging, run ./install-device.sh with the staging build. Then exercise the publish flow on a session that previously triggered the toast.

Prerequisites

  • Staging build (branch fix/publish-toast-diagnostic merged into staging) installed on Carl's iPhone.
  • A session ready to publish that has previously fired the “raw-archive upload incomplete” toast. (Any recent publish session on staging should reproduce.)
  • Best signal: Xcode → Window → Devices and Simulators → iPhone CHM → Open Console, filter on Runner and watch for _uploadRawArchives FAILED lines. Alternative: long-press the failed-pill on the session card to read {Documents}/conversion_error.log (PR #213 sheet).

1 · Toast copy reads cleanly

  • 1. Toast wording is the new copy. Publish a session that triggers the “optional treatment files” toast. Expected: the SnackBar shows “Published. Some optional treatment files are still processing — retry publish in a minute to backfill the rest. Line treatment is live now.” The parenthetical “(raw-archive upload incomplete)” should NOT appear. If you still see “raw-archive upload incomplete” in parentheses, the copy didn't deploy.
  • 2. Toast does NOT fire when all uploads succeed. Publish a session that has previously fully published (i.e. all rawArchiveUploadedAt stamps are present). Expected: only the green “Published ✓” toast — no “optional treatment files” SnackBar layered on top. If you see the long warning toast on a clean publish, the optional-artifacts gate regressed.

2 · Diagnostic debugPrint surfaces the failing variant

  • 3. Open Console.app, attach to iPhone CHM, filter Runner. Then publish the failing session. Expected: at least one log line of the shape _uploadRawArchives FAILED kind=<kind> path=<path> local=<abs> exists=<bool> — with one of these kind values: raw_archive_upload_failed, raw_archive_segmented_upload_failed, raw_archive_photo_upload_failed, raw_archive_color_thumb_failed, raw_archive_mask_upload_failed. Copy the full line into the notes field. If no FAILED line shows but the toast still fires, the diagnostic patch didn't catch the path (escalate).
  • 4. Read the exists= bool. If exists=false, the local file is missing on disk — we'll need to investigate why the variant didn't get produced by the conversion pipeline (likely the photo _thumb_color.jpg not generated). If exists=true, the file IS on disk but the upload is rejected server-side (RLS, content-type, wrong path) — we'll need to check the Supabase storage logs. Either way, paste the exists= finding into the note.
  • 5. Long-press the failed-pill on the session card. The conversion-error sheet (PR #213) should open with the last 5 entries from conversion_error.log. The new raw_archive_* kinds should render as chips alongside any conversion failures. Skip if not applicable — this surface only fires if there's an existing failed-pill on the card. If the long-press worked but the new raw_archive_* meta isn't there, that means loudSwallow's log_error RPC pathway didn't pick up the new meta fields (separate issue).

3 · No behavioral regression

  • 6. Publish still succeeds end-to-end. Despite the optional-artifact failure, the “Published ✓” toast should still appear, the plan version should bump, the line-drawing should be available on the public web player (open in Safari + verify the published plan loads). The diagnostic patch is observation-only — it must not break the publish path.
  • 7. Re-publish 30 seconds later: the “backfill in a minute” promise holds. Hit publish again on the same session. Expected: previously-uploaded variants are skipped (look for _uploadRawArchives: skip lines in console); the still-failing variant fires the same FAILED line again (deterministic). If the second publish “succeeds” with no toast, the failure was actually transient (good news). If it fails the same way, the failure is genuinely deterministic and the diagnostic line is the smoking gun.
