Device QA — Default treatment swap + soft fallback to line

Two changes in one PR.

(1) Default-default flip. New captures + legacy NULL preferred_treatment rows now render as B&W (grayscale), not line. The capture-time write of 'grayscale' shipped in PR #309 (2026-05-12); PR-B flips the READ-time fallback in 4 places (Dart resolver, Studio viewer, MiniPreview, web JS treatmentFromWire) so legacy NULL rows from pre-PR-#309 captures also render as B&W.

(2) Soft fallback on the player. When the practitioner preferred B&W / Original but the client only consented to line, get_plan_full returns grayscale_url=null while keeping line_drawing_url populated. Today the no-fallback resolver renders a coral "treatment not available" placeholder. After PR-B that case silently renders line drawing — no label, no warning. The no-fallback principle still covers the FILE-MISSING case (line drawing also absent / URL expired).

Out of scope (saved for PR-C): publish progress sheet, atomic-publish failure UX, partial-success toast removal.
