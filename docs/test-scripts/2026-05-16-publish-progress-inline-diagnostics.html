2026-05-16 · Publish progress sheet inline diagnostics — THE third fix

Device QA — Inline diagnostic views kill modal-stacking tap-dead

Architectural fix, not a patch. This is the third attempt at the publish-failure diagnostic tap-dead.

PR #357 set useRootNavigator: true on the diagnostic sheets to layer them over the progress sheet. Tap fired the haptic + callback but the child modal pushed onto a navigator that already owned a modal — rejected.

PR #362 dropped that flag. Same outcome: _publishFromToolbar / _reopenPublishSheet routed the diagnostic opens through rootScaffoldMessengerKey.currentContext, which puts the child sheet on the root navigator while the parent lived on Studio's local navigator. Different navigators → invisible child.

This PR (third fix): no second modal. PublishProgressSheet owns an internal _view state with three variants (progress / failureDetail / errorDetail) and swaps its body via setState when the practitioner taps "Show which files" or "Show error details". Back chevron at top-left of each detail view returns to the progress view. One modal route. No navigator-scope guesswork. The whole bug class is gone, not patched.

Files refactored: publish_progress_sheet.dart (view state + body builders + drops onShowFailureDetails / onShowErrorDetails from public API), upload_diagnostic_sheet.dart (chrome stripped, exposes UploadDiagnosticBody), upload_error_details_sheet.dart (same refactor → UploadErrorDetailsBody), studio_mode_screen.dart (4 caller blocks removed — the rootScaffoldMessengerKey routing that caused the bug).

Stacks on: 5e107f8 (PR #362 publish-diagnostic-surface).

Out of scope: web-player changes (R-10 doesn't apply — this is mobile chrome only), Supabase RPC changes, atomic-publish flow (PR-C + PR #362 still authoritative for what triggers the diagnostic state).

Sensitive zone: publish flow chrome — touches app/lib/screens/studio_mode_screen.dart publish path. Carl review before merge per feedback_sensitive_code_review_before_merge.md.
