2026-04-21 · Wave 7 — Silent-failure observability

Device QA — Wave 7

Purpose: verify the 3-item observability MVP works on device: _loudSwallow persists to local log + error_logs table, boot-time self-check screen catches the bleeders (vault, client_id, storage policy), and the daily publish_health cron fires a WhatsApp ping when archive gap > 0.

In scope: _loudSwallow helper, error_logs table writes, lint rule on bare catch blocks, boot-time Diagnostics screen, signed_url_self_check() RPC, publish_health view, WhatsApp cron.

Out of scope: full Result<T, E> boundary refactor (deferred follow-up), Sentry/Datadog, modal error dialogs (explicitly rejected). Design reference: docs/design-reviews/silent-failures-2026-04-20.md.

Prerequisites

  • Wave 7 PR merged; Supabase migration creating error_logs table, signed_url_self_check RPC, and publish_health view applied.
  • Lint hook wired into pre-commit (or CI) so new bare catch (e) {} blocks fail the build.
  • CallMeBot WhatsApp skill credentials wired to the pg_cron / Edge Function that runs the daily publish-health drift check.

1 · _loudSwallow writes to diagnostics.log + error_logs

Every route that used to catch (e) { debugPrint(e); } now routes through _loudSwallow. Local log rotates at 5 MB; online fire-and-forget POSTs to error_logs when severity ≥ warn.

  • 1. Force a tier-C failure on device (e.g. airplane mode ON → publish a plan, let raw-archive upload fail) — a JSON line lands in {Documents}/diagnostics.log with feature, error, context_json, severity fields.
  • 2. Airplane mode OFF → trigger another tier-C failure — a row appears in error_logs table within a few seconds (query via Supabase dashboard or a helper RPC). Includes practice_id and feature.
  • 3. Fill diagnostics.log above 5 MB (or simulate via a test harness) — file rotates; old log renamed or truncated; new writes land in fresh file.
  • 4. Grep the installed Flutter bundle sources (or inspect via git grep -nP) — no bare catch (e) {} blocks remain. Every catch either calls _loudSwallow, re-throws, or returns a typed Result.

2 · Boot-time self-check / Diagnostics screen

App launches run a green/red invariant matrix. Failed probes block the offending action (e.g. publish disabled if vault is bad) and show a practitioner-readable banner.

  • 5. Open the Diagnostics screen (Settings → Diagnostics or equivalent) — a matrix of probes renders with coral/sage dots per check: signed_url_self_check, schema contract hash, SUPABASE_URL + anon key presence, current practice consistency.
  • 6. Temporarily corrupt the vault secret (ask Carl to set it to a wrong value in Supabase vault) → cold-launch app — signed_url_self_check probe goes red, a banner on Home shows "Video playback misconfigured — tap for details", and publish is blocked.
  • 7. Restore the vault secret → relaunch — probe flips green, banner clears, publish re-enables. No manual cache clear needed.
  • 8. On the Diagnostics screen, tap "Send diagnostics to support" — generates a share sheet with the current diagnostics.log attached, redacting any secrets. Share via Messages / AirDrop / email works.

3 · publish_health view + daily WhatsApp cron

publish_health counts plans published in last 24h vs distinct (plan_id, exercise_id) rows in raw-archive. archive_gap > 0 fires a WhatsApp alert via CallMeBot.

  • 9. Query select * from publish_health where day > now() - interval '7 days' via the Supabase SQL editor — returns one row per day with plans_published, archives_present, and archive_gap columns populated.
  • 10. Force an archive gap: publish a plan, then manually delete the raw-archive objects for one exercise — next daily cron run (or manual trigger) sends Carl a WhatsApp message listing the gap: plan id, exercise id, practice id.
  • 11. Restore the archive objects (re-upload via portal admin or re-publish) → next cron run → no WhatsApp ping (zero gap). Carl's phone stays quiet.
  • 12. Severity=error rows appearing in error_logs within the last hour also trigger a separate rollup WhatsApp ping (scoped to Carl, not every practice owner) with a capped summary.
Any Tier-A failure that sneaks past the boot probe is a critical fail on this wave — those are the bleeders the whole design is meant to catch.
