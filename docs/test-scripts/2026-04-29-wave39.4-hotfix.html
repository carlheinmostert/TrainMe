Wave 39.4 · mobile hotfix QA

Sticky pacing + photo refresh + reachability + UTC timestamps

Five mobile fixes from Wave 39 device-QA feedback. (A) Sticky per-client defaults (Wave 8 / Wave 39) extend to two more PACING fields: video_reps_per_loop + inter_set_rest_seconds. They now propagate the same way reps / sets / hold / prep already do — same in-memory overlay, same RPC. (B) Photo last-exercise refresh got a proper fix: the conversion listener now unconditionally reloads the session from SQLite on every done event and merges the conversion payload over the matching row, so spinners clear in-place even when the parent SessionShellScreen's async refresh races us. didUpdateWidget also gained a guard: parent re-pushes are only adopted when they're strictly fresher (more rows or higher conversion-status rank) than local state. (C) Reachability drop-pill latch persists until the practitioner taps the pill again (or backgrounds the app). Card taps and upward scrolls no longer reset it — latching a list to thumb-zone and immediately tapping a card is the whole point. (D) When latched, the dropped list is now clipped above the toolbar's top edge so content can't bleed over Back / Preview / Publish. (E) Dart wire timestamps emit UTC at every Postgres-bound site in upload_service.dart (4 sites: 2x plans.created_at, 1x plans.sent_at, 1x plan_issuances.issued_at). Local log files keep host time.

A · Sticky pacing fields propagation

  • 1. REPS IN VIDEO override propagates to next capture on same client.
    1. Open Client A. Create a new session. Capture a video (long-press shutter, ~3s). Wait for conversion to finish.
    2. Open the exercise card's PACING accordion. Set REPS IN VIDEO = 5 (the default is 3).
    3. Swipe right to Capture mode. Long-press shutter and record exercise 2.
    4. Swipe back to Studio. Open exercise 2's PACING accordion. REPS IN VIDEO reads 5, not 3.
    5. DURATION PER REP for exercise 2 should reflect video_duration / 5 (since per-rep derives from the new sticky default).
  • 2. Inter-set rest override propagates to next capture on same client.
    1. On Client A, in any captured exercise, open PACING. Set POST REP REST (the inter-set rest field) = 30s.
    2. Capture another exercise on Client A.
    3. Open exercise 3's PACING accordion. POST REP REST reads 30s, not the system default of 15s.
  • 3. Pacing defaults are isolated per client.
    1. Carry the Client A overrides from items 1 + 2 (REPS IN VIDEO = 5, POST REP REST = 30s).
    2. Back out to Home. Open Client B (or create a new one). New session. Capture an exercise.
    3. Open the new exercise card's PACING accordion. REPS IN VIDEO reads 3 (system default), POST REP REST reads 15s. The Client A overrides do NOT leak.
    4. Switch back to Client A → new session → capture → PACING shows REPS IN VIDEO = 5, POST REP REST = 30s. Client A's map preserved through the round-trip.

B · Photo last-exercise refresh (proper fix)

  • 4. Photo as LAST exercise — spinner clears in-place within ~1.5s WITHOUT leaving Studio.
    1. Create a new session on any client. Swipe right to Capture mode.
    2. Tap the shutter (short tap = photo). Photo captures. Swipe LEFT back to Studio immediately.
    3. Studio shows the photo card with a converting spinner. Wait. Within ~1.5s the spinner crossfades to a green check (conversion done). The card is now interactive — tap it → full-screen viewer opens.
    4. Critical: do NOT navigate away and come back. The fix is that the spinner clears WITHOUT needing to leave Studio + reopen. This was the regression in Wave 39 that we're re-fixing with the unconditional SQLite reload.
  • 5. Multiple rapid photo captures — all clear without intervention.
    1. In a session, swipe to Capture and take 4 photos in rapid succession (one tap, swipe-left-swipe-right-tap, etc — or just tap shutter 4x in Capture).
    2. Swipe back to Studio. All 4 cards visible with spinners initially, then within a few seconds all 4 settle to green checks.
    3. Tap each card → full-screen viewer opens for each. None stuck on converting.
  • 6. Mid-list photo capture still updates correctly (regression check).
    1. In Studio, drag a photo card so it's NOT the last in the list (e.g. capture a video after, or reorder).
    2. Capture another photo — doesn't matter where in the list. Spinner appears, then clears within ~1.5s. Position in the list is correct.
    3. Capture sequence: photo → video → photo → video. All four cards eventually settle to a green check. None get stuck on converting.

C · Reachability latch + clip

  • 7. With pill latched: tapping any exercise card does NOT reset the latch.
    1. Open a session with 3+ exercises so the reachability pill appears bottom-right. Tap the pill once — list slides down ~50%, pill arrow flips up.
    2. Tap any exercise card body. The card EXPANDS (DOSE / PACING accordions become available) — but the list STAYS dropped. Pill arrow remains flipped up.
    3. Edit a field on the expanded card. Latch persists; list does not snap up.
    4. Collapse the card (tap it again, or tap another card). Latch still persists.
  • 8. With pill latched: scrolling the list up does NOT reset the latch.
    1. Latch the list down via the pill.
    2. Scroll the list up by a small flick or drag (~30pt). The list scrolls within its dropped window, but the dropped position itself does not snap back to the default.
    3. Pill arrow stays flipped up.
  • 9. Tap pill again — latch releases, list slides back up.
    1. From the latched-down state, tap the pill once.
    2. List slides back up to the default position. Pill arrow flips back to down.
    3. Sole way to manually un-latch (other than backgrounding the app).
  • 10. Background & resume: latch is released on resume.
    1. Latch the list down via the pill.
    2. Send the app to the background (home gesture). Wait a couple seconds. Reopen homefit.
    3. Studio is back at the default (un-latched) position. Pill arrow shows down.
  • 11. When latched, dropped list content is clipped above the toolbar's top edge.
    1. Latch the list down. Look at the bottom of the screen, just above the StudioBottomBar (Back / Preview / Publish row).
    2. No exercise card content is visible BELOW the toolbar's top edge or behind the toolbar buttons. The list ends cleanly at the toolbar boundary.
    3. The reachability pill itself is still visible bottom-right ~12pt above the toolbar — only the LIST content is clipped, not the pill.
    4. Repeat with various session lengths (3, 6, 10 exercises) to confirm clipping holds at all densities.

D · UTC timestamps (mobile half)

  • 12. Publish a plan — new plan_issuances.issued_at is in UTC.
    1. Open or create a session with at least 1 exercise. Hit Publish (consumes 1 credit).
    2. Note the WALL-CLOCK time at moment of publish (e.g. 14:32 SAST).
    3. Open the portal → Audit. Newest plan.publish row shows the matching wall-clock time in the viewer's local TZ. (Audit page renders UTC → viewer TZ; if the row had been written in SAST it would render 2h ahead.)
    4. If you're overseas (in a non-SAST TZ), confirm the dashboard / audit timestamp matches YOUR local clock at the moment of publish.
  • 13. plans.sent_at + plans.created_at are in UTC.
    1. Either run a SQL query against the latest publish (select sent_at, created_at from plans where id = '...') or eyeball via the portal's plan detail page.
    2. Both timestamps look correct relative to the publish wall-clock time. If they were 2h ahead, the SAST regression has not been fixed.
    3. If you don't have SQL access: focus on item 12's portal Audit row — that's sourced from plan_issuances.issued_at which uses the same DateTime.now().toUtc() emitter.

Out of scope — reminder

  • The portal-side dashboard timestamp item (P1) is owned by the portal-agent slice of Wave 39.4. Carl's wave-39 results JSON flagged it; track it in the portal's separate test script.
  • No backfill of historical plan_issuances.issued_at rows. Carl explicitly said no backfill — the SAST drift on existing rows stays as a known historical artefact.
  • Local log timestamps (raw_archive_error.log, capture-screen haptic ticks, diagnostics dump) intentionally keep host time. Those files are read by humans on the device, not by Postgres / portal.
  • The unified-preview WebView bridge (unified_preview_scheme_bridge.dart) emits ISO timestamps for in-process JSON handoff to the local web player. No server round-trip; no UTC normalisation needed.
  • Mobile player's "Show me" consent override is unchanged. Wave 39.4 doesn't touch playback.
