Device QA — Hero crop auto-pick + Demo-tab drag override

Bug. The lobby's 1:1 hero crop defaulted to 0.5 (centre vertical band) whenever exercises.hero_crop_offset was NULL. For portrait videos that meant the middle of the frame became the hero — often catching whatever's behind the practitioner (a TV in Carl's QA capture) rather than the person themselves.

Fix. Two-part. (1) When the native segmenter runs to produce the B&W practitioner thumbnail, it now ALSO computes the soft-mask's free-axis centroid (vertical for portrait sources, horizontal for landscape) and returns it in the platform-channel response. Dart adopts that value as the default hero_crop_offset on every fresh capture + every re-scrub. (2) The Demo-tab vertical drag override (HeroCropViewport in the editor sheet) already shipped — it commits a manual value with a 200ms debounce and persists straight to SQLite. When the practitioner later re-scrubs the temporal hero star, auto-pick re-fires on the new frame and overrides the manual drag (simpler than tracking auto vs manual separately; the user just picked a new frame so they'll likely re-drag anyway).

R-10 parity. No web changes touched. web-player/hero_resolver.js + lobby.js already read heroCropOffset from get_plan_full's exercise payload — they'll inherit the new auto-pick value automatically on the practitioner's next publish.

Prerequisites

  • Branch feat/hero-crop-auto-and-drag built + installed on Carl's iPhone via ./install-device.sh staging. Web-player needs no redeploy (consumers already read hero_crop_offset).
  • A client with at least one published portrait video plan. Ideally: the same capture Carl used to spot the TV-as-hero bug, so the auto-pick fix is directly comparable to the old centre crop.
  • For the override checks: a fresh video capture where the auto-pick centroid lands somewhere the practitioner disagrees with (no person in the dominant frame, e.g. scrub the temporal star to a transition frame).

Auto-pick on first publish · THE primary win

  • 1. Capture a new portrait video (yourself or Carl on a treadmill / mat). Wait for conversion to finish (B&W thumbnail renders). Open the editor sheet's Demo tab. The 1:1 hero viewport is positioned over the person, not the centre vertical band. THE fix. device
  • 2. Publish the plan. Open staging.session.homefit.studio/p/<planId> in Safari. The lobby's 1:1 hero tile frames the person, not whatever's mid-frame. R-10 parity proof — no web changes needed; the new auto-pick value just flowed through. device
  • 3. Capture a landscape video (rotate phone before recording). Confirm the hero crop window slides HORIZONTALLY to centre on the person, not stuck at the left edge or middle. (Auto-pick honours the source orientation; portrait → vertical offset, landscape → horizontal offset, matches the resolver's free-axis convention.) device

Override via Demo-tab drag

  • 4. On any portrait video exercise, open the editor sheet → Demo tab. Drag the hero viewport VERTICALLY. The 1:1 box translates live as you drag (the sage dim around it follows the new position; no flicker). Release the drag. device
  • 5. Republish the plan. Open the lobby in Safari. The hero tile reflects the dragged offset (NOT the auto-pick centroid). device
  • 6. Round-trip persistence. After your drag in item 4, swipe away from the Demo tab to Plan / Notes / Settings, then back to Demo — the dragged offset is still there. Close the editor sheet entirely, reopen the same exercise — still there. device

Re-scrub overrides drag

  • 7. On the exercise you dragged in item 4, move the temporal Hero star to a NEW frame. Wait ~1s for the regen debounce. The 1:1 viewport jumps to the auto-pick centroid for the new frame, OVERRIDING your manual drag from item 4. (Expected per Phase B in the brief; the user just picked a new frame.) device
  • 8. From item 7's new auto-pick position, drag the viewport again to a different spot. Move the temporal star ONE more time. Again the viewport jumps to the new frame's auto-pick (your second drag is also overridden). Predictable, reversible. device

Regression sweep
