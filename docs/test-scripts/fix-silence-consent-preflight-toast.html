fix/silence-consent-preflight-toast

PR-A made the upload flow consent-agnostic. The client-side preflight RPC validate_plan_treatment_consent is now informational only; consume_credit on the server is the authoritative backstop. When the preflight fails transiently, publish still succeeds — but a noisy SnackBar fires. This patch silences the SnackBar; keeps the diagnostic as a debugPrint.

Test 1 — Normal publish: no consent-skip toast

1. Open a client with a published plan that you can republish.
Open Studio for an existing session that has at least one consented exercise. Make a small edit (rename a circuit, change reps). Tap Publish. Observe the success state.

Expected: "Published" toast (the standard success one). NO toast that says "Published, but treatment-consent pre-check was skipped". The publish flow looks exactly like it did before this change for the happy-path case.

Test 2 — Publish still succeeds (server guard remains authoritative)

2. Confirm the credit was consumed and the plan version bumped.
After Test 1, open the web portal Audit tab. Confirm a fresh plan.published row appears for the session you just republished, with a credit consumption logged immediately before it.

Expected: Audit shows the publish event. Credit balance dropped by 1 (or 2 for > 75min plans). This proves the server-side consume_credit still enforces consent end-to-end even with no client-side preflight noise.

Test 3 — Diagnostic debugPrint still fires (developer-only check)

3. (Optional — only if you can capture device logs.) Confirm the diagnostic line is preserved.
If you have the Mac connected and can stream device logs (Console.app filtering on the app), look for a line containing Consent preflight skipped: when the preflight RPC fails. This only fires when the preflight RPC itself errors (network blip, auth edge); on a clean network you may never see it — that's fine.

Expected: If the preflight ever skips, a debugPrint appears in logs. No UI toast either way.

Skip this test if you can't easily capture device logs; Tests 1 + 2 are the load-bearing ones.
