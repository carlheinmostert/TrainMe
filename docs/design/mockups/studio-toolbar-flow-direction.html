Studio · Toolbar wave — v2 reframe

Workflow toolbar — readiness, not direction

Earlier I proposed a segmented-pill that highlights the "current step". That was wrong — tapping any of these cells navigates away from Studio (or fires a modal), so none of them is ever "currently selected". They're five action launchers: Capture · Adjust · Preview · Publish · Share, each with its own readiness state (already encoded in the existing code via cell opacity: onAdjust != null, hasExercises, canPublish, session.isPublished).

The triangle was layering "direction" on top of "readiness" and the two didn't need each other. Below: four ways to drop the directional cue without losing the readiness signal.

Lifecycle:

Now — directional triangle

For reference. The coral triangle pushes the eye left-to-right, implying a one-way march.

Now

Option 1 — No separator (recommended)

Drop the chrome. Cell-opacity readiness signal carries the whole message; the row is five tappable action chips inside a shared pill.

1
Reads as: a quiet row of action chips. No flow, no chrome, no redundancy.
  • + Smallest possible visual surface — the triangle was redundant on top of the cell-opacity readiness signal.
  • + Easiest to ship — pure delete in studio_bottom_bar.dart (`_Arrow` widget + `_ArrowPainter` go away).
  • + Honest framing: these are 5 launchers, not 5 stages of a tunnel.
  • − Loses any session-progress at-a-glance cue. If you want that signal, layer Option 2 on top.

Option 2 — Hairline progress bar under the pill

One continuous, non-directional cue. A 2px coral track underneath the pill fills as the session matures. Replaces four separators with one piece of chrome.

2
Reads as: "how far along is this session". A glance tells you whether to publish or keep iterating.
  • + Adds genuine information (session maturity) instead of redundant chrome.
  • + Direction-neutral — the bar fills, but the user isn't told which way to walk the steps.
  • − Slight extra pill height (~6px) to host the bar. May fight the bottom card boundary.
  • − Animation can feel "loading-bar-y" if it crosses the boundary on Publish.

Option 3 — Vertical state-dot stacks between cells

Discrete progress per gap. Each separator is a tiny vertical stack of dots that fills as the session moves past that point.

3
Reads as: "what's been touched, what's pending". Decomposes session maturity into four discrete checkpoints.
  • + Highest information density of the four — encodes per-gap progress.
  • + No direction — the dots are vertical stacks, not horizontal arrows.
  • − Novel pattern; needs a moment to learn (or a tooltip on first install).
  • − Custom painter or 16 small DOM nodes. Heavier to maintain than #1 / #2.

Option 4 — Filling connector line between cells

Linear cousin of #3. Each gap is a thin horizontal line that fills coral as the session passes through that step. Lighter weight than the dot stacks.

4
Reads as: a quiet, continuous track that fills behind you as the session matures.
  • + Same per-gap info as #3, lower visual weight.
  • + Echoes the existing coral track motif on the player's progress pill matrix — visually familiar.
  • − Horizontal hairlines might still read as one-way to some users (eye habit). The lack of an arrowhead is the only difference.
  • − Awkward at the ends — first cell has no leading line, last has no trailing line, so the pill looks slightly lopsided.
Recommendation: ship Option 1 (no separator). The cell-opacity readiness signal is already doing the work; deleting the arrow drops genuinely redundant chrome and the pill becomes more honest about what it is — five launchers, not five stages. Keep Option 2 (hairline bar) as a pocket upgrade if a later wave wants session-maturity at a glance.

Implementation: replace _Arrow + _ArrowPainter in app/lib/widgets/studio_bottom_bar.dart:138–221 with a plain SizedBox(width: 4) spacer (or zero if cell padding handles it). Mobile-only change — the workflow pill doesn't appear on the web player.
