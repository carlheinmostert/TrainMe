Practice + My Workouts — one consistent shell

Three top-level surfaces, two visually-distinct controls in one row. The left capsule is Practice mode with its internal Clients / Classes scope split. The right capsule is My Workouts — a separate identity (consumer, not creator) with no practice context and no credits. Practice chip + credits chip are anchored to the left capsule and fade when the user flips to My Workouts. Today, Classes and My Workouts are locked teasers; when each ships, the only thing that changes is the body — the shell stays exactly where it is.

Three in-app states (Clients real / Classes teaser / My Workouts teaser) plus the web-player lobby that funnels casual viewers into the app via an email-magic-link bridge.

03Three things this design pays off

Why the two-capsule layout + email-magic-link bridge is worth committing to now, not later.

The shell never moves

  • When Classes ships: locked teaser body → real list. Segmented control unchanged.
  • When My Workouts ships: locked teaser body → real list. Right capsule unchanged.
  • Practitioners and consumers see different chrome inside the same shell, not different shells.

The conundrum is solved

  • Credits live inside the Clients sub-tab — never floating where they're irrelevant.
  • Practice chip is anchored to the left capsule — fades when the user flips to My Workouts.
  • Two visually-distinct capsules tell the truth that Practice and My Workouts are different identities.

The funnel is real

  • Public link viewer taps "Get the app & import" → email sheet → magic-link → in-app sign-up.
  • Same plan_invitations pipeline serves practitioner-initiated and self-service paths.
  • Public link dies after claim — protects per-plan pricing and POPIA in one stroke.
