Publish flow refactor

Spec + mockups · 2026-05-15 · Awaiting redline

Why this exists

Three load-bearing decisions Carl made on 2026-05-15 that together reshape how publishing works:

  1. B&W becomes the primary treatment (line drawing demotes to one of several practitioner-choice options).
  2. Publish is atomic — every file must land before publish is declared successful; otherwise the whole thing fails and the credit refunds.
  3. Consent is decoupled from upload. Every treatment file uploads on every publish regardless of consent. Consent stays as a pure player-side visibility gate — practitioners can toggle later without re-publishing.

Why the swap

Line drawing was the easy default to ship — it's always consented, the de-identified version is POPIA-friendly by design. But B&W is more visually informative for clients trying to copy a movement: real human form, real space, but still distanced from full-colour identification. Line stays as a treatment option (and the soft consent fallback) but cedes the default slot.

3. Atomic publish (all-or-nothing)

Today's publish flow runs in two phases: line drawing → public bucket (Phase 1, "must succeed for publish to count"), then raw archive variants → private bucket (Phase 2, "best-effort, partial failure surfaces a toast"). The new model collapses both into one mandatory phase.

Today — partial success allowed

  • Phase 1 sends line drawing
  • If Phase 1 succeeds → publish is "successful"
  • Phase 2 tries the rest best-effort
  • Partial failure → toast: "published; some files still processing"
  • Practitioner has to remember to retry to backfill

After — all uploads mandatory

  • Every treatment file uploads
  • If ANY file fails → publish fails
  • Credit refunds silently — never charged on failure
  • Retry button on the progress sheet
  • Once published, all treatments are in the cloud and consent toggles work without re-publishing

4. Consent as visibility gate (not upload gate)

Today, consent gates BOTH the upload (we skip uploading colour if the client hasn't granted colour consent) AND the playback (signed URLs are NULL when consent is revoked). The new model retains only the latter.

Why decouple

If a client revokes consent today and later grants it again, the practitioner has to re-publish the plan to push the missing files. That's wasteful (uploads they already paid for in the credit) and clunky (the practitioner may not even know consent changed). With decoupled uploads, the files are always in the cloud, and consent toggles become a pure player-side switch with zero round-trip.

The distinction matters

The no-fallback principle (ratified 2026-05-14) says: when a treatment file is genuinely missing (URL expired, file never uploaded, etc.) we show a coral "not available" placeholder. That principle stands.

The new soft-fallback rule covers a different case: the file exists, but consent hides it. Line drawing — the universal-consent baseline — is the documented substitute. This is not a silent swap on file failure; it's a deliberate floor for the consent layer specifically.

6. What changes in code

High-level scope — implementation agent will map this to actual diffs.

Mobile (Flutter)

  • Default treatment on new exercises — flip from line → B&W wherever the default is set (capture flow, editor sheet, sticky defaults table).
  • Publish flow (upload_service.dart) — remove consent branching from the raw-archive upload loop; every file uploads every time. Treat each file failure as a hard publish failure.
  • Progress sheet — new bottom sheet widget per the sibling wireframe. Replaces the partial-success toast.
  • Workflow toolbar chip — surfaces "publishing N/M" when the sheet is dismissed; tap to re-open.
  • Failure UX — coral retry button; tap-through to the in-app diagnostic sheet shipped in #345.

Web player

  • Default treatment in the lobby resolver — flip from line → B&W when an exercise has no explicit preferred_treatment.
  • Soft fallback to line — when the player resolver tries the preferred treatment and the signed URL is NULL (consent revoked, not file-missing), fall back to line drawing rather than the coral placeholder. Add a small "Line (showing because B&W is private)" hint? Or silent? — open question.

Backend (Supabase)

  • No schema migration required. The video_consent jsonb on clients already gates the get_plan_full RPC's signed-URL emission. We just remove the upload-side branch.
  • consume_credit + refund — keep the existing atomic pattern. Refund row writes happen on every failure (today they happen only on post-consume failures, but they're already idempotent).

7. Open questions

Q1 · Should soft-fallback be explicit or silent?

When line replaces B&W due to consent, does the client see a small label ("Showing line — B&W is private for this plan"), or is the swap silent and the consent state only readable from the player gear menu? Argument for explicit: clients don't get confused about why the look changed if they previously had B&W. Argument for silent: keeps the experience clean; the practitioner controls the narrative.

Carl's call.

Q2 · Storage cost ceiling

With every publish now uploading every variant regardless of consent, raw-archive storage grows ~3× the line-only baseline (every plan now ships colour + B&W + segmented variants for every exercise). For the typical practitioner running 8-12 plans/month, this is fine. At scale (100+ practitioners), worth knowing the per-plan storage budget. Not blocking — flag for the business case spreadsheet.

Q3 · Per-exercise treatment preference defaults

If preferred_treatment on a new exercise is now B&W instead of line, do existing exercises in mid-edit sessions auto-migrate to B&W, or only newly-captured ones? I'd say: only newly captured (don't surprise the practitioner with treatment changes on plans they've started). Confirm.

8. Phasing & rollout

Suggest landing in three sequential PRs to keep blast radius small:

  1. PR-A · Decouple consent from upload. Just the _uploadRawArchives change — remove the consent branch, always upload. Atomic-failure semantic is the same as today's "any failure halts" but applied across all phases. Low-risk, single file.
  2. PR-B · Default treatment swap. Mobile + web player both flip default from line → B&W. Touches capture flow, editor sheet, sticky defaults, and the web player resolver. Adds the soft-fallback rule. Medium scope; reviewable in 30 min.
  3. PR-C · Progress sheet + atomic publish UX. The new bottom sheet, the workflow chip, the failure retry path, the death of the partial-success toast. Largest of the three; depends on PR-A being in. UI-heavy.
