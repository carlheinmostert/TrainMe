# Legacy schema patches — archive

These `schema_*.sql` files are the **ad-hoc patches that predate the Supabase Branching cutover**. They were applied directly to prod via `supabase db query --linked --file ...` over the period 2026-04-15 → 2026-05-04.

As of 2026-05-11 the canonical schema lives in [`../migrations/20260511065443_baseline.sql`](../migrations/20260511065443_baseline.sql) — a single file generated by introspecting the live prod schema. The baseline supersedes every patch here.

## Do not apply these files

Running any of these against a fresh DB will fail (they assume prior state) or duplicate baseline content (they reference columns the baseline already declares). They exist as **historical reference only** — useful for understanding what changed when, why a specific RPC has its current shape, or what the migration intent was behind a given milestone.

## How to find what you need

| Looking for | Where to look |
|-------------|---------------|
| The full current schema | [`../migrations/20260511065443_baseline.sql`](../migrations/20260511065443_baseline.sql) |
| What's changed since the baseline | Newer files in [`../migrations/`](../migrations/) |
| Historical intent of a specific feature | Search this folder by filename — milestones map to features |

## How new schema changes go in

New migrations live in [`../migrations/`](../migrations/) with timestamp-prefixed filenames (`YYYYMMDDHHMMSS_name.sql`). Use `supabase migration new <name>` to create one, then `supabase db push` (NOT `supabase db query --linked`) to apply via the migration chain. See `docs/CI.md` §8.

## Restoring a file

If we ever need an archived file back in active rotation, `git mv supabase/archive/<file> supabase/` and update the references in `CLAUDE.md`. The git history makes this safe.
